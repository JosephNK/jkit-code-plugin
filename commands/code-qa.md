# QA 문서 생성 커맨드

Task 문서(`TASKS.md`) **또는** Phase 문서(`PHASES.md`)와 프로젝트 아키텍처/컨벤션을 분석하여, **공통 컨텍스트 문서 1개 + 단위별 QA 문서 N개**를 1:1 매칭으로 생성합니다.

기존처럼 통합 `QA.md` / `PHASES-QA.md` 단일 파일을 만든 뒤 슬라이싱하지 않고, 처음부터 단위별 QA 파일을 직접 생성합니다.

입력 문서 종류는 Step 1에서 헤더 패턴으로 자동 감지합니다 — `### Task N` 헤더가 발견되면 **tasks 모드**, `#### Phase N` 헤더가 발견되면 **phases 모드**. 본 커맨드 본문에서 "단위(Unit)"는 감지된 모드에 따라 Task 또는 Phase를 의미하며, 아래 표로 대응 관계를 요약합니다.

| 요소 | tasks 모드 | phases 모드 |
|------|-----------|-------------|
| 입력 문서 기본 경로 | `code-harness/TASKS.md` | `code-harness/PHASES.md` |
| 단위 헤더 (입력) | `### Task N` | `#### Phase N` (정수/소수 모두) |
| 단위별 고유 필드 | 복잡도, type, 의존 디자인 토큰 | 카테고리 |
| 공통 문서 기본 경로 | `code-harness/TASKS-QA-COMMON.md` | `code-harness/PHASES-QA-COMMON.md` |
| 단위별 QA 디렉토리 | `code-harness/tasks-qa/` | `code-harness/phases-qa/` |
| 단위별 QA 파일명 | `Task-N.md` | `Phase-N.md` (`Phase-0.5.md` 등 소수 포함) |

## 사용법

```
/jkit:code-qa <input-file-path>
/jkit:code-qa code-harness/TASKS.md
/jkit:code-qa code-harness/TASKS.md -o code-harness/TASKS-QA-COMMON.md
/jkit:code-qa code-harness/PHASES.md
/jkit:code-qa code-harness/PHASES.md -o code-harness/PHASES-QA-COMMON.md
/jkit:code-qa code-harness/TASKS.md -arch docs/ARCHITECTURE.md -conv docs/CONVENTIONS.md
/jkit:code-qa code-harness/PHASES.md -plan code-harness/PLAN.md
```

## 인자

- `$ARGUMENTS` 첫 번째 값: 입력 문서 경로 (필수) — `TASKS.md` 또는 `PHASES.md`
- `-o <path>`: **공통 문서** 출력 경로 (선택, 기본값은 모드별로 상이 — 위 표 참조). 단위별 QA 디렉토리는 공통 문서 경로에서 자동 도출된다.
- `-arch <path>`: 아키텍처 문서 경로 (선택, 기본값: `docs/ARCHITECTURE.md`)
- `-conv <path>`: 컨벤션 문서 경로 (선택, 기본값: `docs/CONVENTIONS.md`)
- `-plan <path>`: 계획 문서 경로 (선택, 기본값: `code-harness/PLAN.md`) — 디자인 도구/기능 트랙 여부 판정에 사용

## 워크플로우

### Step 1: 인자 파싱 및 문서 로드

1. `$ARGUMENTS`에서 입력 문서 경로, `-o`, `-arch`, `-conv`, `-plan` 옵션 파싱
2. 입력 문서 읽기 — 파일이 없으면 에러 메시지 출력 후 종료
3. **입력 모드 자동 감지** — 입력 문서의 헤더 패턴을 스캔하여 tasks/phases 중 하나로 확정:
   - `### Task N` (N은 정수) 헤더가 1개 이상 발견 → **tasks 모드**
   - `#### Phase N` (N은 정수 또는 소수) 헤더가 1개 이상 발견 → **phases 모드**
   - 둘 다 발견 → 더 많이 발견된 쪽 채택, 경고 출력
   - 둘 다 발견되지 않음 → 에러 메시지 출력 후 종료 ("Task/Phase 헤더를 찾을 수 없습니다. 입력 파일이 `/jkit:code-tasks` 또는 `/jkit:code-phases` 산출물인지 확인하세요.")
4. 모드에 따라 **기본 공통 문서 경로** 및 **단위별 QA 디렉토리** 결정:
   - tasks 모드: 공통=`code-harness/TASKS-QA-COMMON.md`, 단위별=`code-harness/tasks-qa/`
   - phases 모드: 공통=`code-harness/PHASES-QA-COMMON.md`, 단위별=`code-harness/phases-qa/`
   - `-o`로 공통 문서 경로를 명시 지정한 경우:
     - 공통 문서 경로는 그대로 사용
     - 단위별 디렉토리는 `<dirname of -o>/<basename(-o) lowercase, "-qa-common"·".md" 제거>-qa/` 규칙으로 자동 도출
     - 기본 파일명(`TASKS-QA-COMMON.md` / `PHASES-QA-COMMON.md`) 지정 시 위 기본 디렉토리 사용
5. 아키텍처 문서 읽기 (`docs/ARCHITECTURE.md` 또는 `-arch` 경로)
   - 파일이 없으면 경고 출력 후 아키텍처 검증 항목 없이 진행
6. 컨벤션 문서 읽기 (`docs/CONVENTIONS.md` 또는 `-conv` 경로)
   - 파일이 없으면 경고 출력 후 컨벤션 검증 항목 없이 진행

> 이하 본문에서 **"단위(Unit)"** 는 감지된 모드에 따라 Task(tasks 모드) 또는 Phase(phases 모드)를 지칭합니다. 구조적으로 차이가 나는 지점(필드 추출/헤더 레벨/출력 파일명)만 모드별로 분기하며, 검증 카테고리와 QA 본문 규칙은 공통입니다.

### Step 2: 입력 문서 분석

입력 문서에서 다음 정보를 추출 (모드에 따라 필드명이 다를 수 있으며, 공통 필드와 모드별 고유 필드를 구분해 추출):

**공통 추출 항목**:

1. **개요 및 기술 스택**: 프로젝트 목적, 사용 패키지, 프레임워크 제공/별도 추가 분류
2. **아키텍처 요약**: 레이어 구조, 의존성 방향, 데이터 흐름
3. **적용 컨벤션 요약**: 네이밍, DI, 에러처리, 프레임워크 규칙, 테스트 전략
4. **컨벤션 → 단위 적용 매핑 테이블**: 어떤 규칙이 어떤 단위(Task 또는 Phase)에 적용되는지
5. **단위 목록**: 전체 단위 번호, 제목, 선행 관계
6. **단위 상세**: 각 단위의 목표, 작업 내용(tasks 모드) / 영향 파일(phases 모드), 적용 컨벤션, 완료 기준, 참조 산출물

**모드별 고유 필드**:

- **tasks 모드**: 각 Task의 `Phase`(분류 레이블), `복잡도`(High/Medium/Low), `type`(feature/design-system/screen/integration), `디자인 소스`, `의존 디자인 토큰`, `코드 구조` 섹션
- **phases 모드**: 각 Phase의 `카테고리`(기반/도메인/인프라/디자인 시스템/프레젠테이션/통합), `디자인 소스`
  - phases 모드에서는 복잡도/type이 없으므로 **카테고리에서 type을 추론**한다:
    - `카테고리: 디자인 시스템` → `type=design-system`
    - `카테고리: 프레젠테이션` → `type=screen` (해당 Phase가 UI 컴포넌트를 렌더하는 경우, 없으면 `type=feature`)
    - `카테고리: 기반 / 도메인 / 인프라 / 통합` → `type=feature`
  - 복잡도는 영향 파일 수로 추정 (1~3 파일 Low, 4~6 Medium, 7+ High)

**디자인 도구** 및 **기능 트랙 포함 여부** 판정 (두 모드 공통):

- **기본 경로**: `code-harness/PLAN.md` 상단의 `## 트랙 모드` 섹션에서 `디자인 도구`(figma/stitch/none)와 `기능 트랙`(yes/no) 필드를 읽어 확인
- **PLAN이 다른 경로에 있으면**: `-plan <path>` 옵션으로 경로를 받아 해당 파일을 읽는다
- **PLAN-DESIGN.md 자동 로드**: PLAN.md `## 트랙 모드` 섹션의 `디자인 계획 문서:` 포인터를 따라 디자인 계획 문서를 추가로 로드 (디자인 도구 ≠ none 일 때만 의미 있음). Step 3-5 디자인 트랙 검증 규칙 추출 시 Figma URL/Stitch 경로/디자인 토큰/공통 컴포넌트/상태 계약/위젯 교체 가능성은 PLAN-DESIGN.md에서 우선 추출하고, 누락(구버전 단일 파일 PLAN) 시 PLAN.md `## 디자인 트랙` 섹션으로 폴백
- **구버전 호환**: PLAN.md에 `## 트랙 모드` 섹션이 없으면 단일 파일 PLAN으로 간주하고 본문의 `디자인 도구` / `기능 트랙 포함` 필드로 폴백
- **PLAN이 아예 없으면**: 입력 문서에서 **디자인 시스템 단위 존재 여부**만으로 판정한다:
  - tasks 모드: `type=design-system` Task가 1개 이상 존재 → 디자인 도구 non-none
  - phases 모드: `카테고리: 디자인 시스템` Phase가 1개 이상 존재 → 디자인 도구 non-none
  - 두 모드 모두 figma/stitch 구분 불가 시 "unknown-design-tool"로 표기하고 QA 검증 방법은 "수동 확인 필요" 처리
  - 디자인 시스템 단위가 없음 → 디자인 도구 none
- **주의**: `type=screen` / `카테고리: 프레젠테이션`은 디자인 도구 유무와 무관하게 생성되므로 이를 non-none 근거로 사용하지 않는다

### Step 3: 아키텍처/컨벤션 직접 분석

입력 문서(TASKS/PHASES)의 요약에만 의존하지 않고, **원본 아키텍처/컨벤션 문서를 직접 분석**하여 검증 규칙을 추출합니다.
이 단계를 통해 입력 문서 작성 시 누락되었을 수 있는 규칙까지 QA에 포함합니다.

#### 3-1. 아키텍처 문서 분석 → 검증 규칙 추출

아키텍처 문서를 전체 읽고 아래 항목을 추출:

1. **레이어 구조**: 어떤 레이어가 존재하는지 목록화
2. **레이어 → 파일 경로 패턴**: 각 레이어가 매칭되는 폴더 경로/glob. A0 규칙으로 QA 테이블에 포함한다.
3. **의존성 방향 규칙**: 레이어 간 허용/금지되는 import 방향
4. **데이터 흐름 규칙**: Request/Response 경로에서 각 레이어의 역할
5. **레이어별 구현 규칙**: 각 레이어(Entity, Port, UseCase, Adapter, Presentation)의 제약 사항
6. **PLAN의 `## 아키텍처 변경 필요` 섹션 확인**: 존재하면 해당 내용으로 "허용 경로 패턴"을 확장하고 `(PLAN 변경 승인됨)` 으로 표기한다.

추출 결과를 **아키텍처 검증 규칙 테이블**로 정리:

```
| # | 규칙 | 검증 방법 | 적용 대상 레이어 |
|---|------|----------|----------------|
| A0 | 파일 경로가 허용 경로 패턴에 매칭되어야 함 | glob 매칭 / 경로 grep | 전 레이어 |
| A1 | {의존성 방향 규칙} | {import 확인 방법} | {레이어} |
| A2 | {레이어 분리 규칙} | {코드 리뷰 방법} | {레이어} |
| ... | ... | ... | ... |
```

#### 3-2. 컨벤션 문서 분석 → 검증 규칙 추출 (헤딩 단위 강제 순회)

컨벤션 문서의 **모든 `##` 및 `###` 헤딩**을 빠짐없이 순회합니다.

**실행 방법**:
1. 컨벤션 문서에서 모든 `##` / `###` 헤딩을 추출하여 리스트업
2. 각 헤딩의 내용을 읽고 **검증 가능한 규칙**을 추출
3. 조건부 규칙("API 사용 시", "로컬 DB만 사용 시" 등)은 프로젝트 상황과 대조하여 적용/SKIP 판단
4. 추출한 규칙을 **컨벤션 검증 규칙 테이블**로 정리:

```
| # | 헤딩 | 검증 규칙 | 검증 방법 | 적용/SKIP |
|---|------|----------|----------|----------|
| C1 | ## Dependency Rules | {규칙} | {grep/코드 리뷰} | 적용 |
| C2 | ## Naming Convention | {규칙} | {파일명/클래스명 확인} | 적용 |
| C3 | ### Network (API 사용 시) | {규칙} | - | SKIP |
| ... | ... | ... | ... | ... |
```

#### 3-3. 프레임워크 사용 규칙 심층 분석

컨벤션 문서에 프레임워크/라이브러리 사용 규칙이 있으면, 반드시 아래 항목을 추출하여 검증 규칙화:

1. **강제 base class**: 어떤 클래스가 어떤 base class를 상속/확장해야 하는지
2. **대체 컴포넌트**: 기본 컴포넌트 대신 사용해야 하는 커스텀 컴포넌트
3. **전용 소비 패턴**: 프레임워크가 강제하는 소비/구독 패턴
4. **SKIP 규칙**: 프로젝트 상황상 적용하지 않는 규칙 (적용되면 안 되는 것도 검증 대상)

**llms.txt 참조** (컨벤션에 경로가 명시된 경우):

컨벤션 문서에 프레임워크의 llms.txt 경로 안내가 있으면, `code-tasks.md`와 동일한 방식으로 로컬 캐시에서 문서를 읽어 상세 API/패턴을 파악합니다:
1. 컨벤션의 프레임워크 섹션에서 llms.txt 경로 안내를 찾기
2. 안내에 명시된 방법으로 패키지 캐시 경로를 찾기
3. `{package_root}/llms.txt` 및 서브 패키지 문서 읽기
4. 추출 정보: 번들 패키지 목록, public API, base class, 필수 사용 패턴

#### 3-4. 입력 문서와 교차 검증

원본 아키텍처/컨벤션 문서에서 추출한 규칙과 입력 문서(TASKS 또는 PHASES)의 "적용 컨벤션 요약"을 대조하여:

1. **입력 문서에 누락된 규칙 식별**: 원본에 있지만 입력 요약에 없는 규칙 → QA에 추가 반영
2. **불일치 식별**: 입력 문서의 요약과 원본 규칙이 다른 경우 → 공통 문서의 "불일치 경고" 섹션에 기록
3. **SKIP 판정 재검증**: 입력 문서에서 SKIP한 규칙이 프로젝트 상황에 맞게 SKIP되었는지 확인

#### 3-5. 디자인 트랙 검증 규칙 추출 (디자인 도구 ≠ none 일 때만)

디자인 트랙 정보를 기반으로 검증 규칙을 추출합니다. 디자인 도구 = none 이면 이 단계 전체를 생략합니다.

**소스 우선순위**:
1. **PLAN-DESIGN.md** (`## 디자인 트랙`) — Figma URL/Stitch 경로, 디자인 토큰, 공통 컴포넌트, 상태 계약 표, 위젯 교체 가능성을 우선 추출
2. **PLAN.md** (`## 디자인 트랙` — 구버전 단일 파일) — PLAN-DESIGN.md가 없을 때 폴백
3. **TASKS/PHASES 입력 문서** — PLAN/PLAN-DESIGN가 모두 없을 때 design-system 단위 메타로 추론

| # | 규칙 | 검증 방법 | 적용 대상 |
|---|------|----------|----------|
| D1 | 디자인 토큰 사용 (하드코딩 색/spacing 금지) | 코드 grep + 컴포넌트 리뷰 | design-system / screen |
| D2 | 공통 컴포넌트 재사용 (개별 화면에서 임의 스타일 금지) | 위젯 트리 리뷰 | screen |
| D3 | 디자인 소스 일치 — figma 선택 시 Figma Frame의 레이아웃/색/spacing과 구현 일치 | Figma MCP 조회 + 시각 비교 | screen |
| D4 | 디자인 소스 일치 — stitch 선택 시 docs/stitch/*.md의 구성 요소/배치와 구현 일치 | 프롬프트 문서 대조 | screen |
| D5 | 디자인 소스 단일성 — 하나의 screen에 Figma와 Stitch가 동시에 참조되지 않음 | Task 메타 필드 확인 | screen |
| D6 | 위젯 ↔ 상태관리 분리 — 위젯이 PLAN-DESIGN.md `상태 계약` 표의 의존 대상(인터페이스/계약)만 import. 직접 API 호출/repository import/영속화/외부 I/O 금지 | 코드 grep (import 경로) + 상태 계약 표 대조 | screen / 디자인 시스템 |
| D7 | 위젯 교체 가능성 — PLAN-DESIGN.md `위젯 교체 가능성` 섹션의 분리 지점이 코드에 실제로 존재 (인터페이스/View 추상) | 코드 리뷰 + 분리 지점 grep | screen |

**컨벤션에 접근성/반응형/다크모드 규칙이 있으면** D8~ 로 이어서 추가합니다.

### Step 4: QA 검증 카테고리 정의

각 단위(Task 또는 Phase)에 대해 아래 **5개 검증 카테고리** (디자인 도구 ≠ none 일 때는 6개)를 적용합니다.
단위의 Phase/카테고리와 성격에 따라 해당되지 않는 카테고리는 "N/A"로 표시합니다.

#### 4-1. 기능 검증 (Functional)

단위의 **목표**와 **완료 기준**을 기반으로 기능이 정상 동작하는지 확인하는 항목:

- 단위의 "목표" 문장에서 **검증 가능한 동작**을 추출
- 단위의 "완료 기준" 체크리스트를 QA 항목으로 변환
- **입력 → 기대 출력** 형태로 구체화 (가능한 경우)
- 경계값, 에러 케이스 추가 (해당 단위에 비즈니스 로직이 있는 경우)

#### 4-2. 아키텍처 검증 (Architecture)

아키텍처 문서 기반으로 구조적 규칙 준수를 확인하는 항목:

- **파일 경로 레이어 일치 (A0)**: 생성/수정된 모든 파일의 경로가 Step 3-1의 "레이어 → 허용 경로 패턴"에 매칭되는지 (glob 매칭 또는 경로 grep). 매칭되지 않으면 PLAN의 `## 아키텍처 변경 필요` 승인 범위 내인지 추가 확인.
- **의존성 방향**: import가 허용된 방향으로만 이루어지는지
- **레이어 분리**: 해당 레이어에 속하지 않는 코드가 혼입되지 않았는지
- **데이터 흐름**: Request/Response가 정의된 경로를 따르는지
- **Feature 구조**: Full-stack / Presentation-only 타입에 맞는 폴더 구조인지

#### 4-3. 컨벤션 검증 (Convention)

컨벤션 문서와 "적용 컨벤션" 매핑 테이블 기반으로 규칙 준수를 확인하는 항목:

- **네이밍 규칙**: 파일명, 클래스명, 메서드명이 패턴에 맞는지
- **코드 패턴**: DI 등록 방식, 상태관리 패턴, 코드 생성 도구 사용법이 규칙대로인지
- **프레임워크 사용 규칙**: 단위의 "적용 컨벤션" 섹션에 명시된 규칙이 코드에 반영되었는지
  - base class 상속/확장 여부
  - 대체 컴포넌트 사용 여부 (컨벤션에 명시된 래퍼/커스텀 컴포넌트)
  - 전용 소비/구독 패턴 사용 여부
- **SKIP 규칙 미적용 확인**: 입력 문서에서 SKIP으로 판정된 규칙이 코드에 포함되지 않았는지

#### 4-4. 코드 품질 검증 (Quality)

일반적인 코드 품질 기준을 확인하는 항목:

- **빌드 성공**: 프로젝트의 lint/build 명령 통과 (컨벤션 문서에서 빌드 명령 확인)
- **코드 생성**: 코드 생성 도구가 필요한 단위에서 생성 명령 실행 후 에러 없음 (컨벤션의 코드 생성 섹션 참조)
- **파일 크기**: 컨벤션에 명시된 파일 크기 제한 준수
- **에러 처리**: 레이어별 에러 처리 패턴 준수 (bare catch 금지 등)
- **미사용 import/변수**: 불필요한 코드 없음

#### 4-5. 통합 검증 (Integration)

선행 단위와의 연동 및 전체 흐름 동작을 확인하는 항목:

- **선행 단위 호환**: 선행 단위에서 생성된 인터페이스/클래스를 올바르게 사용하는지
- **DI 연결**: Port → Adapter 연결이 정상적으로 등록되었는지 (해당 시)
- **라우팅 연결**: 화면 간 네비게이션이 정상 동작하는지 (해당 시)
- **데이터 흐름**: 레이어 간 데이터 전달이 올바른지

#### 4-6. 디자인 검증 (Design) — type이 design-system 또는 screen 이고 디자인 도구 ≠ none 일 때만

- **디자인 토큰 준수**: 색/타이포/spacing이 디자인 시스템 단위에서 정의한 토큰만 사용하는지 (하드코딩 금지)
- **공통 컴포넌트 재사용**: 버튼/입력/카드 등이 공통 컴포넌트를 통해 구현되었는지
- **디자인 소스 일치 (figma 선택 시)**: Figma Frame의 구조·배치·크기와 구현 결과가 일치하는지 (Figma MCP 사용 가능 시 자동 조회, 없으면 "수동 확인 필요" 표시)
- **디자인 소스 일치 (stitch 선택 시)**: Stitch 프롬프트에 기술된 요소·배치와 구현 결과가 일치하는지
- **단일 디자인 소스**: 단위 메타의 `디자인 소스` 필드가 단일 값인지 (figma와 stitch 혼용 금지)
- **반응형/접근성/다크모드** (PLAN이나 컨벤션에 명시된 경우): 지정된 브레이크포인트, 접근성 요구사항, 다크모드 지원 충족

### Step 5: 단위별 QA 체크리스트 생성

각 단위(Task 또는 Phase)에 대해 Step 4의 카테고리를 적용하여 체크리스트를 생성합니다. 이 단계의 결과는 **독립적인 단위 QA 파일**(Step 7에서 `tasks-qa/Task-N.md` 또는 `phases-qa/Phase-N.md`로 저장)의 본문이 됩니다.

**생성 규칙**:

1. 단위의 "완료 기준"을 기능 검증의 기본 항목으로 포함
2. 단위의 "적용 컨벤션" 섹션에 나열된 규칙을 컨벤션 검증 항목으로 변환
3. 단위의 "코드 구조"(tasks 모드) 또는 "영향 파일"(phases 모드)에서 파일 경로/시그니처를 아키텍처 검증 항목으로 변환
4. 단위의 Phase/카테고리, type(tasks 모드) 또는 추론된 type(phases 모드), 디자인 도구 조합에 따라 적절한 카테고리 가중치 적용:
   - **기반**: 기능 검증 + 코드 품질 중심
   - **도메인**: 아키텍처 검증 + 컨벤션 검증 중심
   - **인프라**: 기능 검증 + 통합 검증 중심
   - **디자인 시스템 (type=design-system)**: 디자인 검증 + 컨벤션 검증 + 코드 품질 중심
   - **프레젠테이션 (type=screen, 디자인 도구 ≠ none)**: 6개 카테고리 전체 적용 (디자인 검증 포함)
   - **프레젠테이션 (type=screen, 디자인 도구 = none)**: 기존 5개 카테고리 (디자인 검증 생략)
   - **통합**: 통합 검증 + 기능 검증 중심

**체크리스트 항목 작성 원칙**:

- 각 항목은 **Yes/No로 판별 가능**한 형태로 작성
- 검증 방법을 명시 (코드 리뷰, 빌드 명령, 실행 테스트 등)
- 관련 파일 경로를 포함 (단위의 "생성/수정 파일 목록" 또는 "영향 파일" 참조)

### Step 6: 회귀 테스트 매트릭스 생성

단위 간 의존 관계(tasks 모드: "선행 Task", phases 모드: "선행 Phase")를 기반으로, 특정 단위 수정 시 재검증이 필요한 단위를 매트릭스로 정리. 이 매트릭스는 **공통 문서(`*-QA-COMMON.md`)에 저장**되어 전체 프로젝트 수준에서 조회됩니다.

**tasks 모드 예시**:

```markdown
| 수정 Task | 영향받는 Task | 재검증 카테고리 |
|-----------|-------------|---------------|
| Task 4 (Entity) | Task 5, 6, 7, 11~14 | 아키텍처, 통합 |
| Task 5 (Port) | Task 6, 7, 9 | 아키텍처, 통합 |
| ... | ... | ... |
```

**phases 모드 예시**:

```markdown
| 수정 Phase | 영향받는 Phase | 재검증 카테고리 |
|-----------|-------------|---------------|
| Phase 1 (도메인) | Phase 1.5, 5, 7, 8 | 아키텍처, 통합 |
| Phase 3.5 (디자인 소스) | Phase 4, 5, 7, 10 | 디자인, 컨벤션 |
| ... | ... | ... |
```

**생성 규칙**:

1. 입력 문서의 선행 관계를 **역방향으로 추적**하여 영향 범위 파악
2. 직접 의존뿐 아니라 **전이 의존(transitive)** 도 포함
3. 영향받는 카테고리는 수정 단위의 성격(카테고리/type)에 따라 결정:
   - Entity/Port / 도메인 변경 → 아키텍처 + 통합 재검증
   - Adapter/Repository / 인프라 변경 → 통합 재검증
   - UI / 디자인 시스템 변경 → 기능 + 컨벤션 + 디자인 재검증
   - DI / 통합 변경 → 통합 재검증
   - 디자인 소스 문서 (phases 모드 Phase 3.5 등) 변경 → 디자인 + 컨벤션 재검증

### 빌드 검증
- [ ] 의존성 설치 명령 성공 (컨벤션에서 확인한 패키지 매니저 명령)
- [ ] 정적 분석/lint 명령 에러 없음 (컨벤션에서 확인한 lint 명령)
- [ ] 코드 생성 명령 성공 (코드 생성 도구 사용 시, 컨벤션에서 확인한 생성 명령)

### 프레임워크 전수 검증

- [ ] 컨벤션에 명시된 base class가 해당 레이어에서 상속/확장되었는지
- [ ] 컨벤션에 명시된 전용 소비/구독 패턴이 사용되었는지

### 의존성 검증

### 다국어 검증 (해당 시)

- [ ] 사용자 노출 문자열에 다국어 처리가 적용되었는지
- [ ] 키 네이밍이 컨벤션에 명시된 패턴을 따르는지

## 8. 검증 우선순위 가이드

### 복잡도별 검증 깊이

| 복잡도 | 검증 깊이 | 비고 |
|--------|-----------|------|
| High | 전 카테고리 심층 검증 + 경계값 테스트 | 핵심 플로우, 복합 로직 |
| Medium | 전 카테고리 기본 검증 | 인프라 구현, DI 등록 등 |
| Low | 기능 + 컨벤션 기본 검증 | 모델 정의, 폴더 생성 등 |

### Phase/카테고리별 집중 검증 포인트

| Phase/카테고리 | 집중 카테고리 | 이유 |
|---------------|-------------|------|
| 기반 설정 | 코드 품질 | 이후 전체 단위에 영향 |
| 도메인 | 아키텍처 | 의존성 방향, 레이어 분리가 핵심 |

- 파일명:
  - tasks 모드: `<slice_dir>/Task-N.md`
  - phases 모드: `<slice_dir>/Phase-N.md` (소수 포함: `Phase-0.5.md`)

code-harness/phases-qa/Phase-1.md

## 주의사항

- 입력 문서(TASKS/PHASES)에 없는 검증 항목을 임의로 추가하지 않음 — 단위의 목표, 완료 기준, 적용 컨벤션에 근거한 항목만 생성
- 아키텍처/컨벤션 문서와 입력 문서 간 불일치가 발견되면 경고 출력 (공통 문서에 "불일치 경고" 섹션 추가)
- 검증 항목은 **실행 가능한 수준**으로 구체적으로 작성 — "코드가 올바른지 확인" 같은 모호한 항목 금지
- 각 체크리스트 항목은 독립적으로 판별 가능해야 함 (다른 항목 결과에 의존하지 않음)
- **디자인 도구 = none 이면 디자인 카테고리(4-6) 전체를 생략** — 불필요한 노이즈 방지
- **디자인 소스 일치 검증**: Figma MCP가 설정되어 있으면 자동 조회, 없으면 항목을 포함하되 검증 방법에 "수동 확인 필요" 명시
- **단위별 파일 독립성**: 각 단위 QA 파일은 단독으로도 사용 가능해야 함. 공통 컨텍스트는 파일 상단의 상대 링크로 참조하되, 단위 자체의 검증 내용은 자기 완결적으로 작성
- **단일 통합 QA 문서 생성 금지**: 기존처럼 `QA.md` / `PHASES-QA.md` 단일 파일을 만들지 않는다. 공통 문서는 항상 `*-QA-COMMON.md` 명칭을 사용한다.
- **슬라이스 스크립트 호출 금지**: `slice-tasks.mjs` / `slice-phases.mjs`는 TASKS/PHASES 원본 문서용이며, QA 생성 단계에서는 사용하지 않는다.
- **모드 일관성**: phases 모드에서는 단위 QA 헤더를 반드시 `#### Phase N` (h4)로, tasks 모드에서는 `### Task N` (h3)로 유지 — harness가 헤더 레벨에 의존함
- **혼용 금지**: 한 번의 실행은 하나의 입력 모드만 처리. tasks-qa/와 phases-qa/를 같은 실행으로 동시에 생성하지 않는다.
- **phases 모드에서 복잡도/type 필드**: 입력 PHASES 문서에 원래 없는 필드이므로 Step 2-모드별 고유 필드의 규칙에 따라 추론하며, QA 파일에 "추정"이라고 명시
