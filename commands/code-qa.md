# QA 문서 생성 커맨드

Task 문서(TASKS.md)와 프로젝트 아키텍처/컨벤션을 분석하여, Task별 평가 검증(QA) 체크리스트 문서를 생성합니다.

## 사용법

```
/jkit:code-qa <tasks-file-path>
/jkit:code-qa code-harness/TASKS.md
/jkit:code-qa code-harness/TASKS.md -o code-harness/QA.md
/jkit:code-qa code-harness/TASKS.md -arch docs/ARCHITECTURE.md -conv docs/CONVENTIONS.md
/jkit:code-qa code-harness/TASKS.md -plan code-harness/PLAN.md
```

## 인자

- `$ARGUMENTS` 첫 번째 값: Task 문서 경로 (필수)
- `-o <path>`: 출력 파일 경로 (선택, 기본값: `code-harness/QA.md`)
- `-arch <path>`: 아키텍처 문서 경로 (선택, 기본값: `docs/ARCHITECTURE.md`)
- `-conv <path>`: 컨벤션 문서 경로 (선택, 기본값: `docs/CONVENTIONS.md`)
- `-plan <path>`: 계획 문서 경로 (선택, 기본값: `code-harness/PLAN.md`) — 디자인 도구/기능 트랙 여부 판정에 사용

## 워크플로우

### Step 1: 인자 파싱 및 문서 로드

1. `$ARGUMENTS`에서 Task 문서 경로, `-o`, `-arch`, `-conv`, `-plan` 옵션 파싱
2. Task 문서 읽기 — 파일이 없으면 에러 메시지 출력 후 종료
3. 아키텍처 문서 읽기 (`docs/ARCHITECTURE.md` 또는 `-arch` 경로)
   - 파일이 없으면 경고 출력 후 아키텍처 검증 항목 없이 진행
4. 컨벤션 문서 읽기 (`docs/CONVENTIONS.md` 또는 `-conv` 경로)
   - 파일이 없으면 경고 출력 후 컨벤션 검증 항목 없이 진행

### Step 2: Task 문서 분석

Task 문서에서 다음 정보를 추출:

1. **개요 및 기술 스택**: 프로젝트 목적, 사용 패키지, 프레임워크 제공/별도 추가 분류
2. **아키텍처 요약**: 레이어 구조, 의존성 방향, 데이터 흐름
3. **적용 컨벤션 요약**: 네이밍, DI, 에러처리, 프레임워크 규칙, 테스트 전략
4. **컨벤션 → Task 적용 매핑 테이블**: 어떤 규칙이 어떤 Task에 적용되는지
5. **Task 목록**: 전체 Task 번호, 제목, Phase, 복잡도, 선행 관계
6. **Task 상세**: 각 Task의 목표, 작업 내용, 생성/수정 파일 목록, 적용 컨벤션, 코드 구조, 완료 기준
7. **Task 메타**: 각 Task의 `type`(feature/design-system/screen/integration), `디자인 소스`(Figma Frame ID 또는 Stitch 프롬프트 경로), `의존 디자인 토큰`
8. **디자인 도구** 및 **기능 트랙 포함 여부**:
   - **기본 경로**: `code-harness/PLAN.md` 상단의 `디자인 도구` 필드(figma/stitch/none)와 `기능 트랙 포함` 필드(yes/no)를 직접 읽어 확인
   - **PLAN이 다른 경로에 있으면**: `-plan <path>` 옵션으로 경로를 받아 해당 파일을 읽는다
   - **PLAN이 아예 없으면**: TASKS에서 **`type=design-system` Task 존재 여부**만으로 판정한다:
     - `design-system` Task가 1개 이상 존재 → 디자인 도구 non-none (그러나 figma/stitch 구분은 불가 → "unknown-design-tool"로 표기하고 QA 검증 방법은 "수동 확인 필요" 처리)
     - `design-system` Task가 없음 → 디자인 도구 none
   - **주의**: `type=screen`은 디자인 도구 유무와 무관하게 생성되므로 screen 존재를 non-none 근거로 사용하지 않는다

### Step 3: 아키텍처/컨벤션 직접 분석

Task 문서의 요약에만 의존하지 않고, **원본 아키텍처/컨벤션 문서를 직접 분석**하여 검증 규칙을 추출합니다.
이 단계를 통해 Task 문서 작성 시 누락되었을 수 있는 규칙까지 QA에 포함합니다.

#### 3-1. 아키텍처 문서 분석 → 검증 규칙 추출

아키텍처 문서를 전체 읽고 아래 항목을 추출:

1. **레이어 구조**: 어떤 레이어가 존재하는지 목록화
2. **레이어 → 파일 경로 패턴**: 각 레이어가 매칭되는 폴더 경로/glob. A0 규칙으로 QA 테이블에 포함한다.
3. **의존성 방향 규칙**: 레이어 간 허용/금지되는 import 방향
4. **데이터 흐름 규칙**: Request/Response 경로에서 각 레이어의 역할
5. **레이어별 구현 규칙**: 각 레이어(Entity, Port, UseCase, Adapter, Presentation)의 제약 사항
6. **PLAN의 `## 아키텍처 변경 필요` 섹션 확인**: 존재하면 해당 내용으로 "허용 경로 패턴"을 확장하고 `(PLAN 변경 승인됨)` 으로 표기한다.

추출 결과를 **아키텍처 검증 규칙 테이블**로 정리:

```
| # | 규칙 | 검증 방법 | 적용 대상 레이어 |
|---|------|----------|----------------|
| A0 | 파일 경로가 허용 경로 패턴에 매칭되어야 함 | glob 매칭 / 경로 grep | 전 레이어 |
| A1 | {의존성 방향 규칙} | {import 확인 방법} | {레이어} |
| A2 | {레이어 분리 규칙} | {코드 리뷰 방법} | {레이어} |
| ... | ... | ... | ... |
```

#### 3-2. 컨벤션 문서 분석 → 검증 규칙 추출 (헤딩 단위 강제 순회)

컨벤션 문서의 **모든 `##` 및 `###` 헤딩**을 빠짐없이 순회합니다.

**실행 방법**:
1. 컨벤션 문서에서 모든 `##` / `###` 헤딩을 추출하여 리스트업
2. 각 헤딩의 내용을 읽고 **검증 가능한 규칙**을 추출
3. 조건부 규칙("API 사용 시", "로컬 DB만 사용 시" 등)은 프로젝트 상황과 대조하여 적용/SKIP 판단
4. 추출한 규칙을 **컨벤션 검증 규칙 테이블**로 정리:

```
| # | 헤딩 | 검증 규칙 | 검증 방법 | 적용/SKIP |
|---|------|----------|----------|----------|
| C1 | ## Dependency Rules | {규칙} | {grep/코드 리뷰} | 적용 |
| C2 | ## Naming Convention | {규칙} | {파일명/클래스명 확인} | 적용 |
| C3 | ### Network (API 사용 시) | {규칙} | - | SKIP |
| ... | ... | ... | ... | ... |
```

#### 3-3. 프레임워크 사용 규칙 심층 분석

컨벤션 문서에 프레임워크/라이브러리 사용 규칙이 있으면, 반드시 아래 항목을 추출하여 검증 규칙화:

1. **강제 base class**: 어떤 클래스가 어떤 base class를 상속/확장해야 하는지
2. **대체 컴포넌트**: 기본 컴포넌트 대신 사용해야 하는 커스텀 컴포넌트
3. **전용 소비 패턴**: 프레임워크가 강제하는 소비/구독 패턴
4. **SKIP 규칙**: 프로젝트 상황상 적용하지 않는 규칙 (적용되면 안 되는 것도 검증 대상)

**llms.txt 참조** (컨벤션에 경로가 명시된 경우):

컨벤션 문서에 프레임워크의 llms.txt 경로 안내가 있으면, `code-tasks.md`와 동일한 방식으로 로컬 캐시에서 문서를 읽어 상세 API/패턴을 파악합니다:
1. 컨벤션의 프레임워크 섹션에서 llms.txt 경로 안내를 찾기
2. 안내에 명시된 방법으로 패키지 캐시 경로를 찾기
3. `{package_root}/llms.txt` 및 서브 패키지 문서 읽기
4. 추출 정보: 번들 패키지 목록, public API, base class, 필수 사용 패턴

#### 3-4. Task 문서와 교차 검증

원본 문서에서 추출한 규칙과 Task 문서의 "적용 컨벤션 요약"을 대조하여:

1. **Task 문서에 누락된 규칙 식별**: 원본에 있지만 Task 요약에 없는 규칙 → QA에 추가 반영
2. **불일치 식별**: Task 문서의 요약과 원본 규칙이 다른 경우 → "불일치 경고" 섹션에 기록
3. **SKIP 판정 재검증**: Task 문서에서 SKIP한 규칙이 프로젝트 상황에 맞게 SKIP되었는지 확인

#### 3-5. 디자인 트랙 검증 규칙 추출 (디자인 도구 ≠ none 일 때만)

PLAN/TASKS의 디자인 트랙 정보를 기반으로 검증 규칙을 추출합니다. 디자인 도구 = none 이면 이 단계 전체를 생략합니다.

| # | 규칙 | 검증 방법 | 적용 대상 |
|---|------|----------|----------|
| D1 | 디자인 토큰 사용 (하드코딩 색/spacing 금지) | 코드 grep + 컴포넌트 리뷰 | design-system / screen |
| D2 | 공통 컴포넌트 재사용 (개별 화면에서 임의 스타일 금지) | 위젯 트리 리뷰 | screen |
| D3 | 디자인 소스 일치 — figma 선택 시 Figma Frame의 레이아웃/색/spacing과 구현 일치 | Figma MCP 조회 + 시각 비교 | screen |
| D4 | 디자인 소스 일치 — stitch 선택 시 docs/stitch/*.md의 구성 요소/배치와 구현 일치 | 프롬프트 문서 대조 | screen |
| D5 | 디자인 소스 단일성 — 하나의 screen에 Figma와 Stitch가 동시에 참조되지 않음 | Task 메타 필드 확인 | screen |

**컨벤션에 접근성/반응형/다크모드 규칙이 있으면** D6~ 로 이어서 추가합니다.

### Step 4: QA 검증 카테고리 정의

각 Task에 대해 아래 **5개 검증 카테고리**를 적용합니다.
Task의 Phase와 성격에 따라 해당되지 않는 카테고리는 "N/A"로 표시합니다.

#### 4-1. 기능 검증 (Functional)

Task의 **목표**와 **완료 기준**을 기반으로 기능이 정상 동작하는지 확인하는 항목:

- Task의 "목표" 문장에서 **검증 가능한 동작**을 추출
- Task의 "완료 기준" 체크리스트를 QA 항목으로 변환
- **입력 → 기대 출력** 형태로 구체화 (가능한 경우)
- 경계값, 에러 케이스 추가 (해당 Task에 비즈니스 로직이 있는 경우)

#### 4-2. 아키텍처 검증 (Architecture)

아키텍처 문서 기반으로 구조적 규칙 준수를 확인하는 항목:

- **파일 경로 레이어 일치 (A0)**: 생성/수정된 모든 파일의 경로가 Step 3-1의 "레이어 → 허용 경로 패턴"에 매칭되는지 (glob 매칭 또는 경로 grep). 매칭되지 않으면 PLAN의 `## 아키텍처 변경 필요` 승인 범위 내인지 추가 확인.
- **의존성 방향**: import가 허용된 방향으로만 이루어지는지
- **레이어 분리**: 해당 레이어에 속하지 않는 코드가 혼입되지 않았는지
- **데이터 흐름**: Request/Response가 정의된 경로를 따르는지
- **Feature 구조**: Full-stack / Presentation-only 타입에 맞는 폴더 구조인지

#### 4-3. 컨벤션 검증 (Convention)

컨벤션 문서와 "적용 컨벤션" 매핑 테이블 기반으로 규칙 준수를 확인하는 항목:

- **네이밍 규칙**: 파일명, 클래스명, 메서드명이 패턴에 맞는지
- **코드 패턴**: DI 등록 방식, 상태관리 패턴, 코드 생성 도구 사용법이 규칙대로인지
- **프레임워크 사용 규칙**: Task의 "적용 컨벤션" 섹션에 명시된 규칙이 코드에 반영되었는지
  - base class 상속/확장 여부
  - 대체 컴포넌트 사용 여부 (컨벤션에 명시된 래퍼/커스텀 컴포넌트)
  - 전용 소비/구독 패턴 사용 여부
- **SKIP 규칙 미적용 확인**: Task 문서에서 SKIP으로 판정된 규칙이 코드에 포함되지 않았는지

#### 4-4. 코드 품질 검증 (Quality)

일반적인 코드 품질 기준을 확인하는 항목:

- **빌드 성공**: 프로젝트의 lint/build 명령 통과 (컨벤션 문서에서 빌드 명령 확인)
- **코드 생성**: 코드 생성 도구가 필요한 Task에서 생성 명령 실행 후 에러 없음 (컨벤션의 코드 생성 섹션 참조)
- **파일 크기**: 컨벤션에 명시된 파일 크기 제한 준수
- **에러 처리**: 레이어별 에러 처리 패턴 준수 (bare catch 금지 등)
- **미사용 import/변수**: 불필요한 코드 없음

#### 4-5. 통합 검증 (Integration)

선행 Task와의 연동 및 전체 흐름 동작을 확인하는 항목:

- **선행 Task 호환**: 선행 Task에서 생성된 인터페이스/클래스를 올바르게 사용하는지
- **DI 연결**: Port → Adapter 연결이 정상적으로 등록되었는지 (해당 시)
- **라우팅 연결**: 화면 간 네비게이션이 정상 동작하는지 (해당 시)
- **데이터 흐름**: 레이어 간 데이터 전달이 올바른지

#### 4-6. 디자인 검증 (Design) — type이 design-system 또는 screen 이고 디자인 도구 ≠ none 일 때만

- **디자인 토큰 준수**: 색/타이포/spacing이 디자인 시스템 Task에서 정의한 토큰만 사용하는지 (하드코딩 금지)
- **공통 컴포넌트 재사용**: 버튼/입력/카드 등이 공통 컴포넌트를 통해 구현되었는지
- **디자인 소스 일치 (figma 선택 시)**: Figma Frame의 구조·배치·크기와 구현 결과가 일치하는지 (Figma MCP 사용 가능 시 자동 조회, 없으면 "수동 확인 필요" 표시)
- **디자인 소스 일치 (stitch 선택 시)**: Stitch 프롬프트에 기술된 요소·배치와 구현 결과가 일치하는지
- **단일 디자인 소스**: Task 메타의 `디자인 소스` 필드가 단일 값인지 (figma와 stitch 혼용 금지)
- **반응형/접근성/다크모드** (PLAN이나 컨벤션에 명시된 경우): 지정된 브레이크포인트, 접근성 요구사항, 다크모드 지원 충족

### Step 5: Task별 QA 체크리스트 생성

각 Task에 대해 Step 4의 카테고리를 적용하여 체크리스트를 생성합니다.

**생성 규칙**:

1. Task의 "완료 기준"을 기능 검증의 기본 항목으로 포함
2. Task의 "적용 컨벤션" 섹션에 나열된 규칙을 컨벤션 검증 항목으로 변환
3. Task의 "코드 구조"에서 클래스/메서드 시그니처를 아키텍처 검증 항목으로 변환
4. Task의 Phase와 type, 디자인 도구 조합에 따라 적절한 카테고리 가중치 적용:
   - **기반 설정**: 기능 검증 + 코드 품질 중심
   - **도메인**: 아키텍처 검증 + 컨벤션 검증 중심
   - **인프라**: 기능 검증 + 통합 검증 중심
   - **디자인 시스템 (type=design-system)**: 디자인 검증 + 컨벤션 검증 + 코드 품질 중심
   - **프레젠테이션 (type=screen, 디자인 도구 ≠ none)**: 6개 카테고리 전체 적용 (디자인 검증 포함)
   - **프레젠테이션 (type=screen, 디자인 도구 = none)**: 기존 5개 카테고리 (디자인 검증 생략)
   - **통합**: 통합 검증 + 기능 검증 중심

**체크리스트 항목 작성 원칙**:

- 각 항목은 **Yes/No로 판별 가능**한 형태로 작성
- 검증 방법을 명시 (코드 리뷰, 빌드 명령, 실행 테스트 등)
- 관련 파일 경로를 포함 (Task의 "생성/수정 파일 목록" 참조)

```markdown
### Task N: {제목}

**Phase**: {Phase}
**복잡도**: {복잡도}
**type**: {type}

#### 기능 검증 (Functional)
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...

#### 아키텍처 검증 (Architecture)
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...

#### 컨벤션 검증 (Convention)
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...

#### 코드 품질 검증 (Quality)
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...

#### 통합 검증 (Integration)
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...

#### 디자인 검증 (Design) — type이 design-system 또는 screen 이고 디자인 도구 ≠ none 일 때만 포함
- [ ] {검증 항목} — 검증 방법: {방법}
- [ ] ...
```

### Step 6: 회귀 테스트 매트릭스 생성

Task 간 의존 관계를 기반으로, 특정 Task 수정 시 재검증이 필요한 Task를 매트릭스로 정리:

```markdown
| 수정 Task | 영향받는 Task | 재검증 카테고리 |
|-----------|-------------|---------------|
| Task 4 (Entity) | Task 5, 6, 7, 11~14 | 아키텍처, 통합 |
| Task 5 (Port) | Task 6, 7, 9 | 아키텍처, 통합 |
| ... | ... | ... |
```

**생성 규칙**:

1. Task 문서의 "선행 Task" 관계를 **역방향으로 추적**하여 영향 범위 파악
2. 직접 의존뿐 아니라 **전이 의존(transitive)** 도 포함
3. 영향받는 카테고리는 수정 Task의 성격에 따라 결정:
   - Entity/Port 변경 → 아키텍처 + 통합 재검증
   - Adapter 변경 → 통합 재검증
   - UI 변경 → 기능 + 컨벤션 재검증
   - DI 변경 → 통합 재검증

```markdown
## 5. 공통 검증 항목

프로젝트 전체에 걸쳐 반복 확인해야 하는 항목.
아래 카테고리별로 **컨벤션 문서와 아키텍처 문서에서 추출한 프로젝트별 규칙**을 나열합니다.

### 빌드 검증
- [ ] 의존성 설치 명령 성공 (컨벤션에서 확인한 패키지 매니저 명령)
- [ ] 정적 분석/lint 명령 에러 없음 (컨벤션에서 확인한 lint 명령)
- [ ] 코드 생성 명령 성공 (코드 생성 도구 사용 시, 컨벤션에서 확인한 생성 명령)

### 프레임워크 전수 검증
컨벤션 문서의 "프레임워크/라이브러리 사용 규칙"에서 추출한 강제 규칙을 나열:
- [ ] 컨벤션에 명시된 대체 컴포넌트가 기본 컴포넌트 대신 사용되었는지
- [ ] 컨벤션에 명시된 base class가 해당 레이어에서 상속/확장되었는지
- [ ] 컨벤션에 명시된 전용 소비/구독 패턴이 사용되었는지

### 의존성 검증
- [ ] 패키지 매니페스트에 프레임워크 제공 패키지가 중복 추가되지 않음
- [ ] import/의존성 방향이 아키텍처 규칙을 위반하지 않음

### 다국어 검증 (해당 시)
컨벤션 문서에 다국어(i18n) 규칙이 있는 경우에만 포함:
- [ ] 사용자 노출 문자열에 다국어 처리가 적용되었는지
- [ ] 키 네이밍이 컨벤션에 명시된 패턴을 따르는지

## 6. 검증 우선순위 가이드

### 복잡도별 검증 깊이

| 복잡도 | 검증 깊이 | 비고 |
|--------|-----------|------|
| High | 전 카테고리 심층 검증 + 경계값 테스트 | 핵심 플로우, 복합 로직 |
| Medium | 전 카테고리 기본 검증 | 인프라 구현, DI 등록 등 |
| Low | 기능 + 컨벤션 기본 검증 | 모델 정의, 폴더 생성 등 |

### Phase별 집중 검증 포인트

| Phase | 집중 카테고리 | 이유 |
|-------|-------------|------|
| 기반 설정 | 코드 품질 | 이후 전체 Task에 영향 |
| 도메인 | 아키텍처 | 의존성 방향, 레이어 분리가 핵심 |
| 인프라 | 기능 + 통합 | 실제 데이터 연동 정확성 |
| 프레젠테이션 | 컨벤션 | 프레임워크 사용 규칙 집중 |

```

### Step 8: Self-Validation (저장 전 필수 검증)

QA 문서를 저장하기 **전에** 아래 체크리스트를 순회합니다.
하나라도 실패하면 해당 항목을 수정한 뒤 다시 검증합니다.

#### 8-1. 완전성 검증

- [ ] Task 문서의 모든 Task에 대해 QA 체크리스트가 생성되었는가?
- [ ] 각 Task의 "완료 기준" 항목이 빠짐없이 기능 검증에 포함되었는가?
- [ ] 각 Task의 "적용 컨벤션" 항목이 빠짐없이 컨벤션 검증에 포함되었는가?

#### 8-2. 컨벤션 매핑 교차 검증

- [ ] Task 문서의 "컨벤션 → Task 적용 매핑 테이블"에 나열된 모든 규칙이, 해당 Task의 QA 컨벤션 검증에 반영되었는가?
- [ ] 프레젠테이션 레이어 Task의 QA에 프레임워크 base class/컴포넌트 검증 항목이 있는가?

#### 8-3. 회귀 매트릭스 검증

- [ ] 회귀 테스트 매트릭스가 Task 간 의존 관계와 일치하는가?
- [ ] 전이 의존이 누락 없이 포함되었는가?

#### 8-4. 원본 문서 교차 검증

- [ ] Step 3에서 추출한 아키텍처 검증 규칙이 QA의 아키텍처 검증 카테고리에 모두 반영되었는가?
- [ ] Step 3-1의 **A0 규칙 (파일 경로 레이어 일치)**이 모든 Task의 아키텍처 검증에 포함되었는가?
- [ ] Step 3에서 추출한 컨벤션 검증 규칙 중 "적용"으로 판정된 규칙이 QA의 컨벤션 검증에 모두 반영되었는가?
- [ ] Step 3-4에서 식별된 "불일치"가 QA 문서의 "불일치 경고" 섹션에 기록되었는가?
- [ ] Step 3-3에서 추출한 프레임워크 사용 규칙이 해당 Task의 QA 항목에 반영되었는가?

#### 8-5. 검증 방법 구체성

- [ ] 모든 체크리스트 항목에 검증 방법이 명시되어 있는가?
- [ ] 검증 방법이 "확인" 같은 모호한 표현이 아닌, 구체적 동작(코드 리뷰, grep, 빌드 명령, 실행 테스트 등)으로 되어 있는가?

#### 8-6. 디자인 검증 반영 확인 (디자인 도구 ≠ none 일 때만)

- [ ] 디자인 시스템 Phase(type=design-system)의 모든 Task에 디자인 검증 카테고리가 생성되었는가?
- [ ] 모든 screen Task(type=screen)의 QA에 "디자인 소스 일치" 항목이 존재하는가?
- [ ] 각 screen Task의 QA에 "단일 디자인 소스"(figma와 stitch 혼용 금지) 검증 항목이 존재하는가?
- [ ] "디자인 토큰 준수" 항목이 하드코딩 스타일 검출 방법(grep 등)과 함께 명시되어 있는가?
- [ ] Step 2에서 추출한 `의존 디자인 토큰`이 해당 screen Task의 디자인 검증에 반영되었는가?

#### 8-7. 모드 일관성 검증

- [ ] 기능 트랙 = no 인 경우, 기반/도메인/인프라 Task에 대한 QA 블록이 없는가?
- [ ] 디자인 도구 = none 인 경우, 디자인 검증 카테고리(4-6)가 어떤 Task에도 포함되지 않았는가?
- [ ] QA 요약 테이블의 "디자인" 열이 디자인 도구 상태(포함/생략)에 맞게 구성되었는가?

### Step 9: 저장 및 Task 슬라이싱

#### 9-1. QA 문서 저장

Step 8 검증을 모두 통과한 QA 문서를 출력 경로에 저장합니다.

#### 9-2. Task 단위 슬라이스 생성

저장 직후 QA 문서를 Task 단위로 분할하여 슬라이스 파일을 생성합니다.
이 슬라이스는 `/jkit:code-harness` 실행 시 Evaluator에 전달되어
**QA.md 전체를 매 라운드 읽지 않도록** 토큰 사용량을 줄입니다.

**슬라이스 출력 디렉토리** (입력 파일 경로 기반 규칙):

```
slice_dir = <dirname of OUTPUT> / <basename(OUTPUT) lowercase, .md 제거> /
```

- `code-harness/QA.md` → `code-harness/qa/`
- `path/to/MyQA.md` → `path/to/myqa/`

**실행**:

```bash
JKIT_DIR=$(jq -r '.plugins["jkit@jkit"][0].installPath' ~/.claude/plugins/installed_plugins.json)
$JKIT_DIR/scripts/slice-tasks.sh --mode full <OUTPUT> <slice_dir>
```

스크립트가 자동으로 수행하는 작업:
1. `### Task N` 헤더 기준으로 분할
2. QA 문서는 **full 모드**로 슬라이스 생성
   - 공통 헤더(개요·QA 요약 테이블 등)
   - 해당 Task QA 체크리스트
   - 공통 푸터(회귀 테스트 매트릭스, 공통 검증 항목, 검증 우선순위 가이드)
3. footer 영역의 `### Task N ...` 추가 주의 블록은 full 모드 특성상 모든 QA 슬라이스에 포함
4. 슬라이스 첫 두 줄에 출처 메타 주석 삽입
5. 입력에 더 이상 존재하지 않는 Task 슬라이스 자동 제거

**예시 출력**:
```
code-harness/qa/Task-1.md
code-harness/qa/Task-2.md
...
```

> 슬라이싱 실패 시 사용자에게 경고하되 QA 문서 저장 자체는 유지한다. harness 실행 시 슬라이스가 없으면 원본 QA 문서를 fallback으로 사용한다.

### Step 10: 완료 보고

- 생성된 QA 문서 파일 경로 출력
- 생성된 슬라이스 디렉토리 경로 + 슬라이스 개수 출력
- Task 총 개수, 카테고리별 검증 항목 수 요약 출력
- 검증 항목이 가장 많은 Task (= 가장 중점적으로 QA해야 할 Task) 하이라이트
- Step 8 검증 결과 요약 (전체 통과 / 수정된 항목)
- 커밋은 하지 않음 — 사용자가 직접 커밋
- **다음 단계 안내**: `/jkit:code-harness "Task 1"` 실행으로 Generator↔Evaluator 피드백 루프 시작 가능 (`--loop` 옵션으로 자동 반복 가능)

## 주의사항

- Task 문서에 없는 검증 항목을 임의로 추가하지 않음 — Task의 목표, 완료 기준, 적용 컨벤션에 근거한 항목만 생성
- 아키텍처/컨벤션 문서와 Task 문서 간 불일치가 발견되면 경고 출력 (QA 문서에 "불일치 경고" 섹션 추가)
- 검증 항목은 **실행 가능한 수준**으로 구체적으로 작성 — "코드가 올바른지 확인" 같은 모호한 항목 금지
- 각 체크리스트 항목은 독립적으로 판별 가능해야 함 (다른 항목 결과에 의존하지 않음)
- **디자인 도구 = none 이면 디자인 카테고리(4-6) 전체를 생략** — 불필요한 노이즈 방지
- **디자인 소스 일치 검증**: Figma MCP가 설정되어 있으면 자동 조회, 없으면 항목을 포함하되 검증 방법에 "수동 확인 필요" 명시
