# 계획 문서 생성 커맨드

`ecc:plan` 스킬을 래핑하여 요구사항 정리 → 리스크 평가 → 단계별 구현 계획을 수립하고, 결과를 두 파일로 분리 저장합니다:

- `code-harness/PLAN.md` — **기능 계획** (정본). 트랙 모드 헤더 포함.
- `code-harness/PLAN-DESIGN.md` — **디자인 계획**. 디자인 도구 ≠ none 인 경우에만 생성.

생성된 `PLAN.md`는 이후 `/jkit:code-tasks` / `/jkit:code-phases` / `/jkit:code-qa` 커맨드의 진입점이며, 디자인 도구 ≠ none 일 때 다운스트림 커맨드는 PLAN.md 헤더의 `디자인 계획 문서:` 포인터를 따라 `PLAN-DESIGN.md`를 추가로 로드합니다.

## 사용법

```
/jkit:code-plan <요구사항 또는 설명>
/jkit:code-plan 로그인 화면에 2FA 추가
/jkit:code-plan 로그인 화면에 2FA 추가 -o code-harness/PLAN.md
/jkit:code-plan 로그인 화면에 2FA 추가 -od code-harness/PLAN-DESIGN.md
/jkit:code-plan 로그인 화면에 2FA 추가 -arch docs/ARCHITECTURE.md -conv docs/CONVENTIONS.md
/jkit:code-plan 로그인 화면 디자인을 새 Figma로 교체 --design-only
```

## 인자

- `$ARGUMENTS` 본문: 계획을 수립할 요구사항/설명 (필수)
- `-o <path>`: 기능 계획 출력 파일 경로 (선택, 기본값: `code-harness/PLAN.md`)
- `-od <path>`: 디자인 계획 출력 파일 경로 (선택, 기본값: `code-harness/PLAN-DESIGN.md`). 디자인 도구 = none 이면 사용되지 않음.
- `-arch <path>`: 아키텍처 문서 경로 (선택, 기본값: `docs/ARCHITECTURE.md`)
- `-conv <path>`: 컨벤션 문서 경로 (선택, 기본값: `docs/CONVENTIONS.md`)
- `--design-only`: **디자인 트랙만 재작성**. 기존 PLAN.md를 읽기 전용으로 로드하여 기능 트랙과 상태 계약을 보존하고, PLAN-DESIGN.md만 새로 저장. 디자인 도구를 figma↔stitch로 교체하거나 같은 도구 내에서 새 시안으로 교체할 때 사용. PLAN.md가 없거나 기존 디자인 도구가 `none`이면 에러.

## 사전 조건

이 커맨드는 **`ecc:plan` 스킬에 의존**합니다. `everything-claude-code` 플러그인이 설치되어 있어야 동작합니다.

`ecc:plan` 스킬이 사용 가능한지 확인한 뒤 진행합니다. 사용 불가 시 사용자에게 안내하고 종료합니다.

## 워크플로우

### Step 1: 인자 파싱 및 사전 조건 확인

1. `$ARGUMENTS`에서 `-o <path>`, `-od <path>`, `-arch <path>`, `-conv <path>`, `--design-only` 옵션을 분리:
   - `OUTPUT` ← `-o` 값, 없으면 `code-harness/PLAN.md`
   - `OUTPUT_DESIGN` ← `-od` 값, 없으면 `code-harness/PLAN-DESIGN.md`
   - `ARCH_PATH` ← `-arch` 값, 없으면 `docs/ARCHITECTURE.md`
   - `CONV_PATH` ← `-conv` 값, 없으면 `docs/CONVENTIONS.md`
   - `DESIGN_ONLY` ← `--design-only` 플래그 존재 여부 (true/false)
2. 나머지 본문을 요구사항 `DESCRIPTION`으로 취급 — 비어 있으면 에러 메시지 출력 후 종료
3. `ecc:plan` 스킬 사용 가능 여부 확인 — 불가 시 "ecc 플러그인 설치 필요" 안내 후 종료
4. **`DESIGN_ONLY = true` 일 때 사전 조건 검증**:
   a. `OUTPUT` (PLAN.md)이 존재해야 함 — 없으면 `"--design-only는 기존 PLAN.md를 보존하며 디자인만 재작성하는 모드입니다. PLAN.md가 없으니 일반 모드로 다시 실행하세요."` 안내 후 종료
   b. PLAN.md `## 트랙 모드` 섹션의 `디자인 도구` 필드를 읽음 (구버전 단일 파일 PLAN인 경우 `## 디자인 트랙` → `디자인 도구:` 필드로 폴백)
      - 값이 `none`이면 `"기존 PLAN.md의 디자인 도구가 none — 갈아끼울 디자인 트랙이 없습니다. 일반 모드로 실행하세요."` 안내 후 종료
   c. PLAN.md에서 다음을 `EXISTING_FEATURE_PLAN`으로 추출 (이후 Step 3 ecc:plan 입력에 첨부):
      - `## 요구사항`, `## 트랙 모드`, `## 프로젝트 제약 (아키텍처/컨벤션)`, `## 아키텍처 변경 필요`(있을 때), `## 기능 트랙`, `## 계획 본문 (기능)` 섹션 전부
   d. 기존 `OUTPUT_DESIGN` (PLAN-DESIGN.md)이 있으면 `EXISTING_DESIGN_PLAN`으로 보관 — 새 디자인 계획의 `상태 계약` 표 "의존 대상"이 기존 인터페이스를 유지하는지 비교 검증에 사용

### Step 2: 아키텍처/컨벤션 로드

계획이 프로젝트 정본 레이아웃과 충돌하지 않도록, ecc:plan 호출 **전에** 제약 조건을 추출한다. 이 단계를 건너뛰면 PLAN이 임의 레이어 구조를 가정해버려 이후 TASKS/Generator까지 오염된다.

1. `ARCH_PATH` 읽기 — 존재하면 아래 항목을 추출하여 `ARCH_CONSTRAINTS`에 요약 저장:
   - 레이어 목록과 각 레이어가 매칭되는 **폴더 경로/glob 패턴** (예: `src/lib/domain/models/**`, `src/app/[locale]/**/_components/`)
   - 레이어 간 허용 의존 방향
   - 데이터 흐름 (Request → Response 경로에서 각 레이어의 역할)
   - 파일이 없으면 `ARCH_CONSTRAINTS = "(문서 없음 — 제약 없이 진행)"` 로 두고 경고 출력
2. `CONV_PATH` 읽기 — 존재하면 아래 항목을 추출하여 `CONV_CONSTRAINTS`에 요약 저장:
   - 폴더/파일 네이밍 규칙
   - 프레임워크/라이브러리 강제 사용 규칙 (base class, 대체 컴포넌트 등)
   - 금지 패턴 (예: 특정 폴더 사용 금지 등)
   - 파일이 없으면 `CONV_CONSTRAINTS = "(문서 없음 — 제약 없이 진행)"` 로 두고 경고 출력
3. 두 문서 모두 없으면 **경고만 출력하고 진행**. 단 이후 Step 5 PLAN.md 템플릿의 "프로젝트 제약" 섹션에 "참조 문서 없음 — 아키텍처 정합성 미검증"을 명시한다.

### Step 3: ecc:plan 스킬 호출

`DESIGN_ONLY` 값에 따라 입력 블록이 달라집니다:
- `DESIGN_ONLY = false` (일반 모드) → **3-A** 지시 블록 사용
- `DESIGN_ONLY = true` (디자인 전용 재작성) → **3-B** 지시 블록 사용

#### 3-A. 일반 모드 입력 블록

Skill 도구로 `ecc:plan`을 호출합니다. `DESCRIPTION` 뒤에 아래 지시 블록을 덧붙여 전달합니다:

```
요구사항을 '기능 트랙'과 '디자인 트랙'으로 나누어 작성해라.

- 기능 트랙 포함 여부(yes/no)를 먼저 명시한다. yes면 비즈니스 로직/API/데이터 모델/상태 흐름을 작성한다. no면 "기존 구현 유지"로 표기한다.
- 디자인 도구를 figma / stitch / none 중 하나로만 선택한다 (혼용 금지).
  - figma 선택: Figma 파일 URL, 화면↔Frame ID 매핑을 작성한다.
  - stitch 선택: Stitch 프롬프트 저장 경로(docs/stitch/*.md)와 화면별 프롬프트 목록을 작성한다.
  - none 선택: 디자인 트랙 전체를 생략 가능.
- 기능 트랙=no 이면서 디자인 도구=none 인 조합은 의미가 없으므로 둘 중 최소 하나는 활성화되어야 한다.
- 디자인 시스템 항목이 필요하면 디자인 토큰 소스와 공통 컴포넌트 목록을 포함한다.

# 디자인 트랙 설계 원칙 (디자인 도구 ≠ none 일 때만 적용)

디자인 트랙이 활성화된 경우 다음 원칙을 따라 작성한다:

- UI(위젯/화면 컴포넌트)와 비즈니스 로직(상태관리)을 분리 설계할 것.
- 위젯은 상태관리 객체의 **구현체가 아닌 인터페이스/계약**만 의존하도록 작성한다.
  위젯이 직접 API를 호출하거나, 데이터 페칭/영속화 로직을 소유하면 안 된다.
- 화면 위젯은 추후 교체 가능하도록 구성한다 — 동일 상태 계약을 만족하는 대체 위젯이
  들어올 수 있어야 한다 (상태관리/라우팅/데이터 흐름은 영향받지 않음).
- 기능 트랙 = yes 인 경우: "상태 계약(State Contract)"을 먼저 정의하고, 디자인 트랙은
  그 계약만 참조한다 (위젯 → 상태관리 단방향 의존).
- 기능 트랙 = no (기존 구현 유지) 인 경우: 기존 상태관리 객체의 공개 인터페이스를
  먼저 식별하고 그 위에 새 위젯을 올린다 — 기존 인터페이스를 깨는 변경 금지.

위 원칙을 PLAN 본문의 디자인 트랙 섹션에 다음 두 항목으로 surface한다:
1. **상태 계약** — 위젯이 구독하는 상태 / 발행하는 이벤트 목록 (표 형식)
2. **위젯 교체 가능성** — 교체 단위 + 교체 시 영향받지 않아야 할 범위

# 프로젝트 제약 조건 (반드시 준수)

아래는 이 프로젝트의 정본 아키텍처/컨벤션에서 추출한 제약이다.
새 레이어/폴더 구조를 **임의로 제안하지 말고** 아래 제약 안에서만 계획을 작성해라.

## 아키텍처 제약
{ARCH_CONSTRAINTS}

## 컨벤션 제약
{CONV_CONSTRAINTS}

## 준수 규칙
- 파일 생성/수정을 수반하는 작업은 반드시 위 아키텍처의 레이어 경로 패턴 안에서 이루어지도록 기술한다.
- 위 제약과 충돌하는 설계가 **꼭 필요하면**, 계획 본문에 `## 아키텍처 변경 필요` 섹션을 분리하여
  (1) 변경 이유, (2) 추가/변경되는 레이어·폴더 경로, (3) 기존 정본에 미치는 영향, (4) 사용자 승인 요청을 명시한다.
- 이 섹션이 없으면 모든 작업은 기존 레이어 안에서 완결되어야 한다.
```

ecc:plan은 다음 절차를 수행합니다:
1. 요구사항 재진술
2. 리스크 및 블로커 식별
3. 단계별 구현 계획 작성
4. **사용자 확인 대기** — 확인 없이 다음 단계로 진행 금지

#### 3-B. `--design-only` 모드 입력 블록

Skill 도구로 `ecc:plan`을 호출하되, `DESCRIPTION` 뒤에 **3-A 대신 아래 블록**을 전달합니다:

```
요구사항: {DESCRIPTION}

이 호출은 디자인 트랙만 재작성하는 모드다. 기존 기능 계획(PLAN.md)은 첨부되어 있으며, 절대 수정하지 마라.

# 기존 기능 계획 (수정 금지 — 참조 전용)

{EXISTING_FEATURE_PLAN}

# 기존 디자인 계획 (있으면 참조 — 상태 계약 의존 대상 유지에 사용)

{EXISTING_DESIGN_PLAN  또는  "(기존 PLAN-DESIGN.md 없음 — 디자인 신규 작성)"}

# 작성 지시

- 기능 트랙은 위 PLAN.md 내용 그대로다. 어떤 변경도 제안하지 마라 (요구사항 재진술/리스크 분석 단계도 기능 트랙은 건드리지 않음).
- 디자인 도구는 사용자가 새로 지시한 값(요구사항 본문 참조) 또는 기존 값을 사용한다. figma↔stitch 교체도 허용.
- 디자인 트랙 본문 (Figma URL/Frame 매핑 또는 Stitch 프롬프트 경로/화면 목록, 디자인 토큰 소스, 공통 컴포넌트 목록)은 새 디자인 의도에 맞춰 새로 작성한다.
- **상태 계약 표의 "의존 대상" 컬럼은 기존 PLAN/PLAN-DESIGN의 의존 대상을 그대로 유지**해야 한다 — 위젯 교체 시 상태관리 영향 금지 원칙 (위 [디자인 트랙 설계 원칙] 동일 적용). 의존 대상을 바꿔야 한다면 그 이유를 응답에 명시하고 사용자 확인을 요청하라.
- 위젯 교체 가능성 섹션도 새 디자인 기준으로 다시 작성한다 (분리 지점은 기존 인터페이스를 유지).
- 위 [디자인 트랙 설계 원칙] 블록과 [프로젝트 제약 조건] 블록을 동일하게 적용한다.
```

[디자인 트랙 설계 원칙]과 [프로젝트 제약 조건]은 3-A의 동일 블록을 그대로 이어 붙입니다.

ecc:plan은 일반 모드와 동일 절차를 따르되, 산출물은 디자인 트랙 한 트랙만입니다.

### Step 4: 사용자 확인 대기

ecc:plan이 제시한 계획에 대해 사용자가 명시적으로 승인("yes", "proceed", "좋아" 등)할 때까지 대기합니다.

사용자가 수정 요청("modify: ...")하면 ecc:plan에 해당 요청을 전달하여 계획을 갱신합니다.

사용자가 취소("no", "cancel")하면 파일 저장 없이 종료합니다.

**계획에 `## 아키텍처 변경 필요` 섹션이 포함된 경우**: 사용자에게 해당 변경의 의도성을 별도로 확인한다. "기존 레이어로도 가능하지 않은가?"를 먼저 제시하고, 사용자가 명시적으로 "변경 필요"라고 응답한 경우에만 진행한다.

### Step 5: PLAN.md / PLAN-DESIGN.md 저장

ecc:plan이 생성한 전체 계획을 **두 파일로 분리**해 저장합니다.

- `OUTPUT` (`code-harness/PLAN.md`) → **기능 계획 정본**. 일반 모드에서는 항상 생성. **`DESIGN_ONLY = true`이면 저장 건너뜀** (기존 파일 유지).
- `OUTPUT_DESIGN` (`code-harness/PLAN-DESIGN.md`) → **디자인 계획**. `디자인 도구 ≠ none` 일 때만 생성. `DESIGN_ONLY = true`인 경우엔 항상 생성.

분리 원칙:
- 트랙 모드 플래그(`기능 트랙`, `디자인 도구`)와 `디자인 계획 문서:` 포인터는 PLAN.md에만 둔다 (다운스트림 진입점).
- 디자인 트랙의 모든 상세(Figma URL, Stitch 경로, 디자인 토큰, 공통 컴포넌트, 상태 계약, 위젯 교체 가능성)는 PLAN-DESIGN.md에만 둔다.
- 두 파일 사이에 정보를 중복하지 않는다 (동기화 부담 회피).

#### 5-3. 모드 조합 규칙

- `기능 트랙 = yes` + `디자인 도구 ≠ none` → PLAN.md + PLAN-DESIGN.md 모두 생성
- `기능 트랙 = yes` + `디자인 도구 = none` → PLAN.md만 생성. PLAN.md의 `디자인 계획 문서:` 필드는 "해당 없음"
- `기능 트랙 = no` + `디자인 도구 ≠ none` → PLAN.md + PLAN-DESIGN.md 모두 생성. PLAN.md의 기능 트랙은 "기존 구현 유지 — 변경 없음", 그러나 기존 상태관리 인터페이스를 식별해 PLAN-DESIGN.md `상태 계약` 표에 의존 대상으로 기록
- `기능 트랙 = no` + `디자인 도구 = none` → **금지** (아무것도 하지 않음). 에러 출력 후 재작성 요청

출력 디렉토리가 없으면 생성합니다. 두 파일 모두 기존 파일이 있으면 덮어쓰기 전 확인.

#### 5-4. 저장 전 자체 검증

**PLAN.md 검증**:
- [ ] PLAN.md 본문이 도입하는 모든 파일/폴더 경로가 `ARCH_CONSTRAINTS`의 레이어 경로 패턴 안에 있는가?
- [ ] 매칭되지 않는 경로가 있다면 `## 아키텍처 변경 필요` 섹션이 작성되고 사용자 승인을 받았는가?
- [ ] 참조 문서가 모두 없는 경우, 제약 섹션에 "참조 문서 없음 — 아키텍처 정합성 미검증"이 명시되었는가?
- [ ] `## 트랙 모드` 섹션의 `기능 트랙` / `디자인 도구` / `디자인 계획 문서` 세 줄이 모두 채워졌는가?
- [ ] 디자인 도구 ≠ none 일 때, `디자인 계획 문서:` 필드가 `{OUTPUT_DESIGN}` 값과 일치하는가?
- [ ] 디자인 도구 = none 일 때, `디자인 계획 문서:` 필드가 "해당 없음"인가?
- [ ] PLAN.md 본문에 디자인 토큰/Figma URL/Stitch 경로/공통 컴포넌트/상태 계약/위젯 교체 가능성 같은 디자인 상세가 새지 않았는가? (PLAN-DESIGN.md로만 보내야 함)

**PLAN-DESIGN.md 검증** (디자인 도구 ≠ none 일 때만):
- [ ] `상태 계약` 표가 작성되고 위젯이 인터페이스/계약만 의존하도록 기술되었는가?
- [ ] `위젯 교체 가능성` 섹션의 교체 단위 / 영향받지 않아야 할 범위 / 분리 지점이 모두 명시되었는가?
- [ ] 위젯이 직접 API 호출 / 데이터 페칭 / 영속화 / 외부 I/O를 수행하는 설계가 포함되어 있지 않은가? (포함되어 있으면 reject)
- [ ] `연결된 기능 계획` 필드가 `{OUTPUT}` 값과 일치하는가?
- [ ] 디자인 도구가 figma인 경우 Figma URL과 Frame 매핑 표가 채워졌는가? stitch인 경우 프롬프트 경로와 화면 목록이 채워졌는가?

**`--design-only` 모드 추가 검증** (`DESIGN_ONLY = true` 일 때만):
- [ ] PLAN.md(`OUTPUT`)가 이번 호출 동안 변경되지 않았는가? (변경 시 reject)
- [ ] 새 PLAN-DESIGN.md의 `상태 계약` 표 "의존 대상" 컬럼이 `EXISTING_FEATURE_PLAN`/`EXISTING_DESIGN_PLAN`의 의존 대상과 일치하는가? 불일치 항목이 있으면 사용자에게 의도성을 확인하고, 명시적 승인 없이 다르면 reject
- [ ] 새 디자인 도구가 기존과 다르면(예: figma → stitch), `EXISTING_DESIGN_PLAN`의 디자인 도구 정본 위치 정책(figma ↔ stitch 혼용 금지)을 준수했는가? — 새 도구 정본 + 기존 도구는 `docs/stitch/history/` 같은 비정본 아카이브로 안내했는가?

하나라도 실패하면 Step 3으로 돌아가 ecc:plan에 재작성을 요청한다.

### Step 6: 완료 보고

**일반 모드** (`DESIGN_ONLY = false`):
- 저장된 파일 경로 출력:
  - PLAN.md: `{OUTPUT}` (항상)
  - PLAN-DESIGN.md: `{OUTPUT_DESIGN}` (디자인 도구 ≠ none 일 때만; 그 외엔 "생성 안 됨 (디자인 도구: none)"으로 표기)
- 모드 요약: `기능 트랙: yes/no` + `디자인 도구: figma/stitch/none`
- 다음 단계 안내: `/jkit:code-tasks {OUTPUT}` 실행으로 Task 문서 생성 가능. 다운스트림 커맨드는 PLAN.md를 진입점으로 사용하며, 디자인 도구 ≠ none 일 때 PLAN.md 헤더의 `디자인 계획 문서:` 포인터를 따라 PLAN-DESIGN.md를 자동 로드합니다.

**`--design-only` 모드** (`DESIGN_ONLY = true`):
- 변경 요약:
  - PLAN.md (`{OUTPUT}`): **변경 없음** (기존 유지)
  - PLAN-DESIGN.md (`{OUTPUT_DESIGN}`): 새로 저장됨
- 디자인 도구 변경 여부: 기존 `{OLD_TOOL}` → 새 `{NEW_TOOL}` (동일하면 "동일")
- 다음 단계 안내 — 다운스트림 재생성 필요:
  - `/jkit:code-tasks {OUTPUT}` (TASKS.md + 슬라이스 갱신)
  - `/jkit:code-phases {OUTPUT}` (PHASES.md + 슬라이스 갱신)
  - `/jkit:code-qa <TASKS_OR_PHASES>` (QA 재생성)
  - 진행 중인 `code-harness` 라운드가 있으면 **`code-harness/harness-state/`를 수동 삭제** 후 재시작 (디자인 시스템/screen Task ID 충돌 방지)

## 주의사항

- 커맨드 내부에서 코드 수정을 하지 않습니다. 계획 수립과 문서 저장만 수행합니다.
- ecc:plan의 판정(리스크, 복잡도 등)을 임의로 변경하지 않고 그대로 보존합니다.
- 기존 `OUTPUT` 또는 `OUTPUT_DESIGN` 파일이 있으면 덮어쓰기 전에 사용자에게 확인을 받습니다.
- **아키텍처 정합성**: PLAN에 포함된 모든 파일/폴더 경로는 `ARCH_PATH` 문서의 레이어 경로 패턴과 일치해야 합니다. 새 경로가 필요하면 `## 아키텍처 변경 필요` 섹션을 작성하고 사용자 승인을 별도로 받으세요. 이 섹션 없이 기존 제약을 벗어난 경로를 제안하는 계획은 저장 전에 수정합니다.
- **디자인 도구 혼용 금지**: figma와 stitch를 동시에 정본으로 두지 않습니다. 예외적으로 Stitch로 초기 시안을 만든 후 Figma로 이관한 경우, 정본은 figma이며 Stitch 프롬프트는 `docs/stitch/history/`에 비정본 아카이브로만 보관합니다 (TASKS.md에서 참조하지 않음).
- **기능 트랙 = no + 디자인 도구 = none 금지**: 아무 작업도 정의되지 않은 상태. 에러 출력 후 재작성 요청.
- **두 파일 분리 정책**:
  - 트랙 모드 플래그(`기능 트랙`, `디자인 도구`)와 `디자인 계획 문서:` 포인터의 **정본은 PLAN.md**. PLAN-DESIGN.md의 `트랙 모드` 섹션은 가독성을 위한 복사본이며, 두 파일이 불일치하면 PLAN.md를 신뢰합니다.
  - 디자인 트랙의 모든 상세(Figma URL/Stitch 경로/디자인 토큰/공통 컴포넌트/상태 계약/위젯 교체 가능성)는 **PLAN-DESIGN.md에만** 둡니다. PLAN.md에 디자인 상세가 새지 않도록 Step 5-4 검증을 통과시킵니다.
- **디자인만 갈아끼우는 워크플로우** (`--design-only` 권장):
  - 1순위: `/jkit:code-plan <새 디자인 요구사항> --design-only` — 기존 PLAN.md를 그대로 두고 PLAN-DESIGN.md만 새로 생성. 상태 계약 표 "의존 대상"은 기존 인터페이스를 강제 유지 (위젯 교체 시 상태관리 영향 금지 원칙).
  - 2순위: PLAN-DESIGN.md를 수동 편집 — 가벼운 수정 시 (단, 자체 검증을 거치지 않으므로 위 분리 원칙은 사용자가 직접 보장).
  - 3순위: `-od code-harness/PLAN-DESIGN-v2.md`로 별도 파일 생성 후 PLAN.md `디자인 계획 문서:` 포인터를 수동 갱신 — A/B 비교나 이력 보존 목적.
  - 어느 경로든 새 PLAN-DESIGN.md가 만들어지면 `/jkit:code-tasks` / `/jkit:code-phases` / `/jkit:code-qa` 재실행이 필요합니다 (다운스트림은 PLAN-DESIGN.md를 자동 로드하지만 TASKS/PHASES/QA 산출물은 수동 재생성).
- **PLAN/TASKS 교체 시 주의**: 기능 구현 완료 후 디자인 트랙을 추가하거나 새 PLAN으로 전환할 때는, `/jkit:code-harness` 실행 전 `code-harness/harness-state/`를 수동으로 삭제하세요. 이전 라운드의 `state.json`(currentTaskId/taskQueue/completedTasks)이 새 Task ID 세트와 충돌할 수 있습니다.
