# Task 생성 커맨드

계획 문서(Plan)와 프로젝트 아키텍처/컨벤션을 분석하여, 구현 순서가 포함된 상세 Task 문서를 생성합니다.

## 사용법

```
/jkit:code-tasks <plan-file-path>
/jkit:code-tasks code-harness/PLAN.md
/jkit:code-tasks code-harness/PLAN.md -o code-harness/TASKS.md
```

## 인자

- `$ARGUMENTS` 첫 번째 값: 계획 문서 경로 (필수)
- `-o <path>`: 출력 파일 경로 (선택, 기본값: `code-harness/TASKS.md`)
- `-arch <path>`: 아키텍처 문서 경로 (선택, 기본값: `docs/ARCHITECTURE.md`)
- `-conv <path>`: 컨벤션 문서 경로 (선택, 기본값: `docs/CONVENTIONS.md`)

## 워크플로우

### Step 1: 인자 파싱 및 문서 로드

1. `$ARGUMENTS`에서 계획 문서 경로, `-o`, `-arch`, `-conv` 옵션 파싱
2. 계획 문서 읽기 — 파일이 없으면 에러 메시지 출력 후 종료
3. 아키텍처 문서 읽기 (`docs/ARCHITECTURE.md` 또는 `-arch` 경로)
   - 파일이 없으면 경고 출력 후 아키텍처 제약 없이 진행
   - 이 경우 Step 8의 아키텍처 기반 검증 항목은 `N/A (아키텍처 문서 없음)`으로 기록
4. 컨벤션 문서 읽기 (`docs/CONVENTIONS.md` 또는 `-conv` 경로)
   - 파일이 없으면 경고 출력 후 컨벤션 제약 없이 진행
   - 이 경우 Step 8의 컨벤션 기반 검증 항목은 `N/A (컨벤션 문서 없음)`으로 기록
5. **라이브러리 llms.txt**: 컨벤션 문서에 llms.txt 경로가 명시된 프레임워크가 있으면 Step 3-2에서 로드

### Step 2: 계획 문서 분석

계획 문서에서 다음 정보를 추출:

1. **프로젝트 개요**: 목적, 대상 사용자, 핵심 기능
2. **기술 스택**: 언어, 프레임워크, 주요 라이브러리
3. **데이터 모델**: 엔티티, 테이블, 관계
4. **기능 목록**: 화면, API, 서비스 등
5. **설계 원칙**: 아키텍처 방식, 분리 기준
6. **기능 트랙 포함 여부**: PLAN의 `## 기능 트랙` 섹션의 `기능 트랙 포함: yes | no` 값 (누락 시 yes로 간주)
7. **디자인 도구**: PLAN의 `## 디자인 트랙` 섹션의 `디자인 도구: figma | stitch | none` 값 (누락 시 none으로 간주)
8. **디자인 트랙 상세**: 디자인 도구 ≠ none 일 때 Figma URL + Frame 매핑 또는 Stitch 프롬프트 경로 목록, 디자인 토큰 소스, 공통 컴포넌트 목록

> **모드 판정**: (기능 트랙 포함, 디자인 도구) 조합으로 아래 Step 4의 Phase 구성을 결정한다.
> **금지 모드**: `기능 트랙 포함: no`이고 `디자인 도구: none`이면 생성할 Task 범위가 없으므로 에러 메시지를 출력하고 종료한다.

### Step 3: 아키텍처/컨벤션 매핑

아키텍처 문서와 컨벤션 문서를 **전체** 분석하여 아래 항목을 추출합니다.
빠뜨린 규칙이 있으면 Task 문서 전체에 영향을 미치므로, 문서의 모든 섹션을 순회해야 합니다.

#### 3-1. 아키텍처 분석

1. **레이어 구조 파악**: 어떤 레이어가 존재하는지 (예: Presentation, Domain, Infrastructure)
2. **레이어 → 파일 경로 패턴 추출**: 각 레이어가 매칭되는 폴더 경로/glob을 명시적으로 추출하여 테이블로 정리한다. Step 8의 경로 검증에서 사용한다.

   ```
   | 레이어 | 허용 경로 패턴 | 비고 |
   |--------|----------------|------|
   | Domain (Models) | src/lib/domain/models/** | |
   | Domain (Ports)  | src/lib/domain/ports/**  | |
   | UI (Shared)     | src/components/**        | |
   | Page            | src/app/**               | |
   ```

3. **의존성 방향 파악**: 레이어 간 허용되는 의존 방향
4. **데이터 흐름 파악**: 요청/응답이 레이어를 통과하는 경로
5. **PLAN의 `## 아키텍처 변경 필요` 섹션 확인**: PLAN에 이 섹션이 있으면 변경되는 레이어/경로를 위 테이블에 `(PLAN 변경 승인됨)` 표기로 추가한다. 없으면 기존 레이어만을 정본으로 삼는다. PLAN이 지시하는 경로가 기존 레이어에 없고 변경 섹션도 없으면 **Task 분해 진행을 중단하고 사용자에게 PLAN 재작성을 요청**한다.

#### 3-2. 컨벤션 분석 — 헤딩 단위 강제 순회

컨벤션 문서의 **모든 `##` 및 `###` 헤딩**을 빠짐없이 순회해야 합니다.

**실행 방법**:
1. 컨벤션 문서에서 모든 `##` / `###` 헤딩을 추출하여 리스트업
2. 각 헤딩을 하나씩 읽으며 아래 6개 카테고리 중 하나로 분류
3. 분류 결과를 테이블로 정리 (누락 방지용 — Task 문서에는 포함하지 않음)

```
| # | 헤딩 | 카테고리 | 프로젝트 적용 여부 | 비고 |
|---|------|---------|-------------------|------|
| 1 | ## Dependency Rules | 의존성 규칙 | 적용 | |
| 2 | ## {framework_name} | 프레임워크 사용 규칙 | 적용 | base class, 컴포넌트 대체 |
| 3 | ### Network (API 사용 시) | 프레임워크 사용 규칙 | SKIP | 로컬 DB only |
| ... | ... | ... | ... | ... |
```

**6개 카테고리**:
1. **의존성 규칙**: 레이어 간 import 제약, cross-feature 규칙
2. **네이밍 규칙**: 파일명, 클래스명, 메서드명 패턴
3. **코드 패턴**: 상태관리, DI 등록, 라우팅, 에러처리 패턴
4. **프레임워크/라이브러리 사용 규칙**: 프로젝트에서 사용하는 프레임워크별 필수 사용법
   - 어떤 base class를 상속/확장해야 하는지 (예: Screen은 `XxxWidget` 상속, Controller는 `BaseController` 확장)
   - 어떤 컴포넌트/유틸을 대신 사용해야 하는지 (예: 기본 컴포넌트 대신 커스텀 래퍼 사용)
   - 조건부 적용 규칙 (예: "API 사용 시", "로컬 DB만 사용하는 경우" 등)
   - **조건부 규칙은 계획 문서의 프로젝트 상황과 대조하여 적용/SKIP 판단**
5. **테스트 전략**: 테스트 도구, 구조, 커버리지 기준
6. **코드 생성**: 코드 생성 도구(annotation processor, code generator 등) 사용 규칙

**"프레임워크/라이브러리 사용 규칙" 카테고리 심층 분석**:

이 카테고리에 해당하는 섹션이 발견되면, 반드시 아래 항목을 추출:
- 해당 프레임워크가 제공하는 **서브 패키지/모듈 목록**
- 각 서브 패키지가 **강제하는 base class, 대체 컴포넌트, 유틸리티**
- 각 규칙의 **적용 조건** (무조건 적용 vs 조건부 적용)
- 해당 프레임워크가 **이미 번들하는 외부 패키지** (→ Step 3-3에서 사용)

**llms.txt 로드 (프레임워크의 상세 문서 확보)**:

컨벤션 문서에 프레임워크의 `llms.txt` 경로가 명시되어 있으면, 해당 경로를 따라가서 로컬 캐시에서 문서를 읽습니다.

1. 컨벤션의 프레임워크 섹션에서 **llms.txt 경로 안내**를 찾기 (예: "LLM Documentation" 섹션)
2. 안내에 명시된 방법으로 **패키지 캐시 경로**를 찾기
   - **Dart/Flutter**: `package_config.json`의 `rootUri`, 또는 `~/.pub-cache/` 검색
   - **Node.js**: `node_modules/{package}/` 경로
   - **기타**: 컨벤션 문서에 명시된 경로를 따름
3. `{package_root}/llms.txt` (메인 인덱스) 읽기
4. 프로젝트에서 사용하는 서브 패키지의 문서도 읽기 (`llms/{package}/llms.txt`)
5. 서브 패키지 문서에 상세 `.md` 링크가 있으면, 로컬 캐시의 동일 경로에서 읽기
6. **추출할 정보**: 번들 패키지 목록, public API, base class, import 경로
7. **llms.txt 안내가 없으면**: 컨벤션 문서와 lock 파일 정보만으로 진행

#### 3-3. 의존성 중복 검사 (CRITICAL — 반드시 실행)

컨벤션에 명시된 프레임워크/라이브러리가 이미 제공하는 패키지를 식별합니다.
이 단계를 건너뛰면 중복 의존성이 Task 문서에 포함됩니다.

**실행 방법**:

1. Step 3-2에서 "프레임워크/라이브러리 사용 규칙" 카테고리로 분류된 섹션에서 **프레임워크 이름** 추출
2. 해당 프레임워크가 **이미 번들하는 패키지**를 다음 순서로 파악:
   - **(a)** lock 파일에서 해당 프레임워크의 transitive dependencies 확인
     - Dart/Flutter: `pubspec.lock`
     - Node.js: `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml`
     - Python: `poetry.lock` / `requirements.txt`
     - 기타: 프로젝트의 lock 파일
   - **(b)** 프레임워크의 패키지 매니페스트 확인 (패키지 캐시 또는 소스)
   - **(c)** 컨벤션 문서 자체에서 "이 패키지가 XX를 제공한다"는 기술이 있는지 확인
   - (a), (b), (c) 중 하나라도 확인되면 "제공됨"으로 판정
3. 계획 문서의 기술 스택에 나열된 패키지를 하나씩 순회하며:
   - 프레임워크가 이미 제공하는 패키지인지 대조
   - 중복이면 "프레임워크 제공" 열에 기록
   - 중복이 아니면 "별도 추가" 열에 기록
4. **결과를 테이블로 출력** — Task 문서의 "2. 기술 스택" 섹션에 포함:

```
### 프레임워크 제공 (별도 추가 불필요)
| 패키지 | 제공 출처 | 용도 |
|--------|-----------|------|

### 별도 추가 필요
| 패키지 | 용도 |
|--------|------|
```

5. **검증**: 최종 의존성 목록에 프레임워크 제공 패키지가 포함되어 있지 않은지 확인

#### 3-4. 컨벤션 → Task 적용 매핑 테이블 작성

추출한 모든 규칙을 Task 작성 시 참조할 수 있도록 매핑 테이블로 정리:

```markdown
| 컨벤션 규칙 | 적용 대상 Task | 구체적 적용 방법 |
|------------|---------------|-----------------|
| {프레임워크 base class 상속 규칙} | {해당 Task} | {구체적 상속/확장 방법} |
| {대체 컴포넌트 사용 규칙} | {해당 Task} | {구체적 사용 방법} |
| {DI/상태관리 등록 규칙} | {해당 Task} | {구체적 등록 방법} |
| ... | ... | ... |
```

이 매핑 테이블은 Task 문서 본문의 "4. 적용 컨벤션 요약"에 포함됩니다.

### Step 4: Task 분해

계획 문서의 기능을 구현 가능한 Task 단위로 분해. 분해 원칙:

1. **레이어 순서** (모드에 따라 동적 구성):
   - 전체 순서: 기반 → 도메인 → 인프라 → **[디자인 시스템]** → 프레젠테이션 → 통합
   - **기능 트랙 = no** 이면 기반/도메인/인프라 Phase를 생략 (기존 구현을 신뢰)
   - **디자인 도구 = none** 이면 [디자인 시스템] Phase를 생략
   - 각 Phase 내 세부:
     - 기반 설정 (의존성, 폴더 구조, DB 스키마 등)
     - 도메인 모델 (Entity, Port/Interface, UseCase/Service)
     - 인프라 구현 (Adapter/Repository, 외부 서비스 연동)
     - 디자인 시스템 (디자인 토큰, 공통 컴포넌트 — 디자인 도구 ≠ none 일 때만)
     - 프레젠테이션 (화면, 상태관리, UI 컴포넌트)
     - 통합 및 테스트
2. **단일 책임**: 한 Task = 한 가지 관심사
3. **독립 실행 가능**: 각 Task 완료 후 빌드/테스트 통과 가능
4. **선행 의존성 명시**: Task 간 순서 관계를 명확히
5. **참조 산출물 선행 보장**: 어떤 Task가 로컬 파일/문서/설정/코드 심볼을 입력으로 참조하면, 그 산출물은 Task 시작 전에 이미 존재하거나 선행 Task에서 생성되어야 한다.
   - 예: 디자인 소스 문서, API 스키마, OpenAPI spec, DB schema, fixture, seed data, 공통 컴포넌트, theme/token 파일, 테스트 설정 파일
   - 후속 Task의 입력으로 쓰이는 문서를 "문서 최종화" 같은 마지막 Task에서 신규 생성하지 않는다. 먼저 필요한 최소 버전을 초기 Task에서 생성하고, 마지막 Task에서는 보강/정렬만 수행한다.
6. **확정 구현 전략만 작성**: Task의 작업 내용/코드 구조/완료 기준에는 "A가 안 되면 B", "문서 확인 후 결정" 같은 미확정 구현 분기를 남기지 않는다. 확인이 필요하면 Task 분해 전에 조사하여 하나의 전략으로 확정하거나, 별도 선행 조사/준비 Task로 분리한다.

#### 모드별 Phase 구성

| 기능 트랙 | 디자인 도구 | 포함 Phase |
|-----------|-------------|-----------|
| yes | none | 기반 → 도메인 → 인프라 → 프레젠테이션 → 통합 |
| yes | figma/stitch | 기반 → 도메인 → 인프라 → **디자인 시스템** → 프레젠테이션 → 통합 |
| no | figma/stitch | **디자인 시스템** → 프레젠테이션(리디자인) → 통합 |
| no | none | 금지 — 에러 메시지 출력 후 종료 |

#### 디자인 시스템 Phase 정의 (디자인 도구 ≠ none 일 때)

0. **디자인 소스 준비** — screen/design-system Task가 참조할 로컬 디자인 문서 또는 디자인 소스 매핑을 먼저 준비
   - figma: PLAN의 Figma URL/Frame 매핑을 Task 문서에 반영. 별도 로컬 매핑 문서를 참조한다면 해당 문서를 먼저 생성
   - stitch: `docs/stitch/{name}.md` 같은 로컬 문서를 디자인 소스로 참조한다면, 디자인 시스템/화면 Task보다 앞선 Task에서 생성하거나 이미 존재함을 확인
1. **디자인 토큰 정의** — 색/타이포/spacing/radius를 프로젝트 토큰 시스템으로 변환
   - figma: Figma Variables 또는 PLAN의 디자인 토큰 소스 참조
   - stitch: PLAN에 명시된 토큰 소스 또는 프롬프트에 기술된 값 참조
2. **공통 컴포넌트 구현** — 버튼/입력/카드 등 재사용 위젯
   - figma: Figma 컴포넌트 1:1 매핑
   - stitch: Stitch 프롬프트에 기술된 컴포넌트 구조 매핑
3. **디자인 검증** (선택) — 토큰/컴포넌트 스냅샷 테스트 또는 스토리북 (프로젝트 지원 시)

### Step 5: Task 상세 작성

각 Task에 대해 다음 항목을 작성:

```markdown
### Task N: {제목}

**Phase**: {해당 Phase 이름}
**복잡도**: High / Medium / Low
**선행 Task**: {의존하는 Task 번호 목록 또는 "없음"}
**type**: feature | design-system | screen | integration
**디자인 소스** (type이 design-system 또는 screen이고 디자인 도구 ≠ none 일 때만 작성, 단일 소스만):
- figma 선택 시: Figma Frame `{node-id}` (PLAN의 매핑 테이블 참조)
- stitch 선택 시: `docs/stitch/{name}.md`
- 로컬 파일을 디자인 소스로 참조하는 경우: 해당 파일이 이미 존재하거나, 이 Task보다 앞선 선행 Task에서 생성되어야 함
- 예외적으로 디자인 소스가 없는 placeholder/design-system Task라면 `디자인 소스: 없음 — {구체적 사유}`를 명시
**의존 디자인 토큰** (type이 screen 일 때만): {디자인 시스템 Task 번호 — 예: Task 7}

**목표**:
{이 Task가 완료되면 달성되는 것 1~2문장}

**작업 내용**:
1. {구체적 작업 항목}
   - 생성/수정할 파일 경로
   - 아키텍처 문서에서 적용할 패턴
   - 컨벤션 문서에서 적용할 규칙
2. ...

**생성/수정 파일 목록**:
- `path/to/file` — {역할 설명}
- ...

**참조 산출물**:
{이 Task가 시작 전에 필요로 하는 로컬 파일/문서/설정/코드 산출물. 없으면 "없음"}
- `path/to/source.md` — {참조 목적, 이미 존재 또는 선행 Task N에서 생성}
- `{SymbolName}` in `path/to/file.ts` — {참조 목적, 선행 Task N에서 생성}

**적용 컨벤션**:
{Step 3-4 매핑 테이블에서 이 Task에 해당하는 규칙 나열}
- {컨벤션 규칙 1} → {구체적 적용 방법}
- {컨벤션 규칙 2} → {구체적 적용 방법}

**코드 구조** (아키텍처/컨벤션 기반):
{해당 Task에서 작성할 코드의 클래스/메서드 시그니처}
{컨벤션에 정의된 패턴을 적용한 코드 예시}
{Step 3에서 파악한 프레임워크 사용 규칙이 반영된 구조}

**완료 기준**:
- [ ] {검증 가능한 기준 1}
- [ ] {검증 가능한 기준 2}
- [ ] 컨벤션 준수 확인 (적용 컨벤션 항목 모두 반영)
- [ ] 빌드 성공 (프로젝트의 lint/build 명령)
```

### Step 8: Self-Validation (저장 전 필수 검증)

Task 문서를 저장하기 **전에** 아래 체크리스트를 순회합니다.
하나라도 실패하면 해당 항목을 수정한 뒤 다시 검증합니다.
단, Step 1에서 아키텍처 문서 또는 컨벤션 문서가 없어 경고 후 진행한 경우, 해당 문서에 의존하는 검증 항목은 실패가 아니라 `N/A`로 기록하고 완료 보고에 포함합니다.

#### 8-1. 의존성 검증

- [ ] Task 문서 "기술 스택" 섹션에 "프레임워크 제공" vs "별도 추가" 테이블이 존재하는가?
- [ ] "별도 추가" 목록에 프레임워크가 이미 제공하는 패키지가 포함되어 있지 않은가?
- [ ] Task 1(의존성 추가)의 패키지 목록이 "별도 추가" 테이블과 일치하는가?

#### 8-2. 컨벤션 헤딩 전수 검증

- [ ] 컨벤션 문서가 없는 경우, 아래 항목을 `N/A (컨벤션 문서 없음)`으로 기록했는가?
- [ ] Step 3-2에서 컨벤션 문서의 모든 `##` 헤딩을 리스트업했는가?
- [ ] 각 헤딩에 대해 적용/SKIP 판단이 내려졌는가?
- [ ] "프레임워크/라이브러리 사용 규칙" 카테고리의 규칙이 Task 문서 "4. 적용 컨벤션 요약"에 포함되어 있는가?

#### 8-3. 프레임워크 사용 규칙 적용 검증

컨벤션에서 추출한 프레임워크 사용 규칙을 하나씩 순회하며, 해당 규칙이 적용되어야 할 Task에 반영되었는지 확인:

- [ ] 프레임워크가 **강제하는 base class**가 있으면 → 해당 Task의 코드 구조에 base class 상속이 명시되어 있는가?
- [ ] 프레임워크가 **대체 컴포넌트**를 강제하면 → 해당 Task의 코드 구조에서 기본 컴포넌트 대신 대체 컴포넌트가 사용되어 있는가?
- [ ] 프레임워크가 **전용 소비 패턴**을 강제하면 → 해당 Task의 코드 구조에 전용 패턴이 명시되어 있는가?
- [ ] **조건부 규칙 중 SKIP한 것**이 Task 문서에 반영되지 않았는가? (적용되면 안 되는 규칙이 포함되어 있지 않은가?)

#### 8-4. Task ↔ 컨벤션 매핑 교차 검증

- [ ] Task 문서 "4. 적용 컨벤션 요약"의 매핑 테이블에 나열된 모든 규칙이, 해당 Task의 "적용 컨벤션" 섹션에도 존재하는가?
- [ ] 프레젠테이션 레이어 Task들의 "코드 구조"에서 프레임워크 base class/컴포넌트가 사용되어 있는가?
- [ ] 각 Task의 "완료 기준"에 "컨벤션 준수 확인" 항목이 포함되어 있는가?

#### 8-5. 구조적 검증

- [ ] Task 간 순환 의존이 없는가?
- [ ] 의존 관계 그래프와 각 Task의 "선행 Task"가 일치하는가?
- [ ] 모든 Task가 완료 시점에 빌드가 깨지지 않는 순서인가?
- [ ] Phase 순서가 Step 4의 모드별 Phase 구성과 일치하는가? 예외가 있으면 Phase를 재분류했거나 명시적 사유가 있는가?
- [ ] 각 Task의 "참조 산출물"이 이미 존재하거나, 해당 Task의 선행 Task에서 생성되는가?
- [ ] 코드 구조/import 예시에 등장하는 로컬 심볼/파일이 어느 Task에서 생성되는지 역추적했고, 직접 사용하는 산출물의 생산 Task가 선행 Task에 포함되어 있는가?
- [ ] 후속 Task의 입력으로 쓰이는 문서/스키마/설정 파일이 마지막 "문서 최종화" 또는 "정리" Task에서 처음 생성되지 않는가?
- [ ] Task의 작업 내용/코드 구조/완료 기준에 미확정 구현 분기("A 실패 시 B", "문서 확인 후 결정" 등)가 남아 있지 않은가?
- [ ] 아키텍처 문서가 없는 경우, 아래 경로/레이어 검증 항목을 `N/A (아키텍처 문서 없음)`으로 기록했는가?
- [ ] **경로 ↔ 레이어 매칭**: 각 Task의 "생성/수정 파일 목록"의 모든 경로가 Step 3-1의 "레이어 → 허용 경로 패턴" 테이블 중 하나에 매칭되는가?
- [ ] **PLAN 변경 승인 반영**: 매칭되지 않는 경로가 있으면 PLAN의 `## 아키텍처 변경 필요` 섹션에 해당 경로가 명시되어 있고 사용자 승인을 받았는가? (승인이 없으면 TASK 경로를 수정하거나 PLAN으로 돌아가 변경 섹션 작성 요청)
- [ ] **PLAN ↔ TASKS 일관성**: PLAN에 `## 아키텍처 변경 필요` 섹션이 있다면 그 내용이 TASKS의 "3. 아키텍처 요약"에도 동일하게 반영되어 있는가?

#### 8-6. 디자인 트랙 검증 (디자인 도구 ≠ none 일 때만)

- [ ] "디자인 시스템" Phase가 Task 목록에 존재하는가?
- [ ] 로컬 디자인 소스 파일을 참조하는 모든 design-system/screen Task에서 해당 파일이 이미 존재하거나 선행 Task에서 생성되는가?
- [ ] 디자인 소스 준비 Task가 필요한 경우, 디자인 시스템/화면 구현 Task보다 먼저 배치되어 있는가?
- [ ] 프레젠테이션 Task(type=screen)의 선행 Task 또는 전이 선행 Task에 "디자인 시스템" Task가 포함되어 있는가?
- [ ] 각 screen Task가 **단일 디자인 소스**(Figma Frame 또는 Stitch 문서/프롬프트)만 참조하는가? (혼용 금지)
- [ ] 디자인 도구 ≠ none 인 모든 design-system/screen Task에 "디자인 소스" 필드가 있거나, `디자인 소스: 없음 — {사유}` 예외가 명시되어 있는가?
- [ ] 디자인 토큰이 특정 Task에 정의되고 이후 screen Task가 이를 재사용하도록 "의존 디자인 토큰" 필드로 명시되어 있는가?

#### 8-7. 모드 일관성 검증

- [ ] 기능 트랙 = no 인 경우, 기반/도메인/인프라 Phase의 Task가 없는가?
- [ ] 디자인 도구 = none 인 경우, design-system/screen Task에 디자인 소스 필드가 없는가?
- [ ] 기능 트랙 = yes 인 경우, 최소 1개의 기반/도메인/인프라/통합 Task가 있는가?

### Step 9: 저장 및 Task 슬라이싱

#### 9-1. Task 문서 저장

Step 8 검증을 모두 통과한 Task 문서를 출력 경로에 저장합니다.

#### 9-2. Task 단위 슬라이스 생성

저장 직후 Task 문서를 Task 단위로 분할하여 슬라이스 파일을 생성합니다.
이 슬라이스는 `/jkit:code-harness` 실행 시 Generator/Evaluator에 전달되어
**TASKS.md 전체를 매 라운드 읽지 않도록** 토큰 사용량을 줄입니다.

**슬라이스 출력 디렉토리** (입력 파일 경로 기반 규칙):

```
slice_dir = <dirname of OUTPUT> / <basename(OUTPUT) lowercase, .md 제거> / 
```

- 단, 출력 파일명이 정확히 `TASKS.md`이면 slice_dir은 `<dirname of OUTPUT>/tasks/`를 사용한다.
- `code-harness/TASKS.md` → `code-harness/tasks/`
- `path/to/MyTasks.md` → `path/to/mytasks/`

**실행**:

실행 전에 `slice-tasks.mjs`의 존재 여부와 실행 권한을 확인합니다. 스크립트가 없거나 실행할 수 없으면 경고를 출력하고 Task 문서 저장은 유지합니다.

```bash
JKIT_DIR=$(jq -r '.plugins["jkit@jkit"][0].installPath' ~/.claude/plugins/installed_plugins.json)
$JKIT_DIR/scripts/slice-tasks.mjs --mode compact <OUTPUT> <slice_dir>
```

스크립트가 자동으로 수행하는 작업:
1. `### Task N` 헤더 기준으로 분할
2. Task 문서는 **compact 모드**로 슬라이스 생성
   - 최소 공통 컨텍스트
   - 해당 Task 상세
   - 해당 Task에 적용되는 컨벤션 행
   - 해당 Task 관련 리스크
   - footer 영역의 `### Task N ...` 추가 주의 블록은 해당 `Task-N.md`에만 포함
3. 슬라이스 첫 줄에 출처 메타 주석 삽입 (`<!-- sliced from ... @ sha ... -->`)
4. 슬라이스 메타에 Task ID와 compact 여부 삽입 (`<!-- task-id: Task N -->`, `<!-- slice-mode: compact -->`)
5. 입력에 더 이상 존재하지 않는 Task 슬라이스 자동 제거 (stale 제거)

**예시 출력**:
```
code-harness/tasks/Task-1.md
code-harness/tasks/Task-2.md
...
```

> 슬라이싱 실패(`### Task N` 헤더 0건 등) 시 사용자에게 경고하되 Task 문서 저장 자체는 유지한다. harness 실행 시 슬라이스가 없으면 원본 Task 문서를 fallback으로 사용한다.

### Step 10: 완료 보고

- 생성된 Task 문서 파일 경로 출력
- 생성된 슬라이스 디렉토리 경로 + 슬라이스 개수 출력
- Task 총 개수, Phase별 분포, 복잡도 분포 요약 출력
- Step 8 검증 결과 요약 (전체 통과 / 수정된 항목)
- 커밋은 하지 않음 — 사용자가 직접 커밋
- **다음 단계 안내**: `/jkit:code-qa {OUTPUT}` 실행으로 QA 체크리스트 문서 생성 가능

## 분해 규칙 상세

### 레이어별 Task 순서 템플릿

아키텍처 문서에서 레이어 구조를 파악한 뒤, 아래 순서를 기본 템플릿으로 적용합니다. 아키텍처에 해당 레이어가 없으면 건너뜁니다.

```
[기반 설정]
  1. 의존성/패키지 추가
  2. 프로젝트 구조 (폴더, 기초 파일)
  3. 데이터 소스 설정 (DB 스키마, API 클라이언트 등)

[도메인 레이어]
  4. 엔티티/모델 정의

```

### Task 크기 기준

- **Small (Low)**: 파일 1~3개, 패턴 적용만으로 완료
- **Medium**: 파일 3~6개, 비즈니스 로직 또는 외부 연동 포함
- **Large (High)**: 파일 6개 이상, 복합 상태 관리 또는 복잡한 UI 인터랙션

### 코드 구조 작성 규칙

- 아키텍처 문서에 코드 예시가 있으면 해당 패턴을 그대로 적용
- 컨벤션 문서에 네이밍/구조 규칙이 있으면 반드시 따름
- **컨벤션에 프레임워크 사용 규칙이 있으면 반드시 해당 base class/컴포넌트/유틸 사용** (프레임워크 기본 컴포넌트로 대체하지 않음)
- **조건부 규칙**("API 사용 시", "로컬 DB만 사용 시" 등)은 프로젝트 상황에 맞는 조건만 적용
- 클래스/메서드/함수 시그니처 수준까지 작성 (구현 본문은 생략 가능)
- Import/의존성 방향이 아키텍처 규칙을 위반하지 않는지 명시
- **프레임워크가 이미 제공하는 패키지는 별도 의존성으로 추가하지 않음** (Step 3-3 결과 반영)

## 주의사항

- 계획 문서에 없는 기능을 추가하지 않음
- 아키텍처/컨벤션 문서와 충돌하는 구조를 만들지 않음
- **컨벤션 문서의 모든 섹션을 빠짐없이 검토** — 특히 프레임워크별 사용 규칙, 조건부 규칙 누락 주의
- **의존성 중복 금지** — 프레임워크가 번들하는 패키지를 별도로 추가하지 않음
- Task 간 순환 의존이 발생하지 않도록 함
- 각 Task 완료 시점에 빌드가 깨지지 않는 순서를 보장
- Task 문서는 "무엇을 어떤 순서로 만들지"에 집중하고, "왜 만드는지"는 계획 문서에 위임
- **디자인 소스 혼용 금지**: 하나의 screen Task가 Figma Frame과 Stitch 프롬프트를 동시에 참조하지 않음
- **모드 일관성**: 기능 트랙 = no 인데 기반/도메인 Task가 생성되거나, 디자인 도구 = none 인데 design-system Task가 생성되면 에러

## Figma MCP 연동 (선택)

디자인 도구가 figma이고 Figma MCP 서버가 설정되어 있으면 Generator가 Task 실행 시 Figma 프레임을 직접 읽어 컴포넌트/레이아웃을 추출할 수 있습니다.

- 설치: Claude Code MCP 설정에 Figma Dev Mode MCP 추가
- TASKS.md에 `디자인 소스: Figma Frame {node-id}`만 기록하면 Generator가 자동으로 조회
- Stitch는 독립 도구 — Stitch에서 생성한 시안은 Figma로 export한 뒤 Frame ID를 기록하는 방식 권장 (혼용 방지)
